# v1166 DPO-lite 偏好微调（边际上升、生成回退的诚实教训）

## 目标

把对齐流水线推进到 SFT 之后的下一步——**偏好微调 DPO（Direct Preference Optimization）这个损失**，但只用在合成的**正确性**信号上（不是人类偏好、不是 RLHF）。从一个**刻意训弱**的 SFT 基座出发，在偏好三元组 (prompt, chosen, rejected) 上优化

```
L = -E[ log σ( β · ( (logp_w - logp_l) - (logp_w^ref - logp_l^ref) ) ) ]
```

其中 `chosen` 是该指令的正确输出，`rejected` 是**同一输入上另一个易混指令**的输出（如 `Rabc=` 的 chosen 是 `cba`，rejected 是排序的 `abc`）。`logp` 是 completion token 的对数概率之和，prompt+pad 用和 SFT 完全一致的掩码屏蔽。

## 设计先于 GPU：对抗式设计评审推翻了讨喜的叙事

开 GPU 前先跑了一轮对抗式设计评审（Workflow，3 个怀疑视角 + 1 个真实 CPU 可行性探针先跑代码）。探针在 8/8 个 (lr×β) 网格里**证伪**了三个天真假设，定位了真正可报告的现象：

1. **「DPO 同时提升边际和 exact-match」——假。** DPO 优化的是**相对边际**，所以 logp(chosen) 会下降、只要 logp(rejected) 下降更快；held-out 生成（exact-match）会**回退到基座以下**。
2. **「DPO 的负样本信号比只用正样本的 SFT 更能压制易混错误」——假（对生成而言）。** 算力对齐的 SFT-on-chosen 控制臂在生成上完胜 DPO。
3. **「参考/KL 项是 DPO 稳定的关键」——本规模下不支持（一个被诚实记录的零结果）。**

## 运行命令

```powershell
python scripts\run_dpo_preference_v1166.py --device cuda --seeds 3
```

## 关键结果（真实 GPU 运行，RTX 4060，3 seeds，4L/128）

`status=pass`、`verdict=dpo_raises_margin_but_generation_regresses`。两个头部指标**绝不合并**：偏好/边际是**优化目标**，held-out exact-match 是**能力指标**，它们会分道扬镳——这个背离本身就是结论。

| 指标（max 预算 1600 forward） | SFT 基座 | DPO（带 ref） | DPO（无 ref） | SFT-on-chosen（控制臂） |
|---|---|---|---|---|
| 平均边际 margin | 14.3 | **86.1**（↑约 6 倍） | — | — |
| 偏好准确率 pref-acc | 0.970 | 0.996（已近天花板） | — | — |
| held-out exact-match | 0.588 | **0.104**（↓回退） | 0.141 | **0.758**（↑） |
| Δlogp(chosen) vs 基座 | — | **−26.7** | — | — |

- **病理确认：** DPO 把边际从 14 拉到 86，但靠的是 logp(chosen) 下跌 −26.7（logp(rejected) 跌得更狠）——生成能力从 0.588 崩到 0.104。
- **杀手级控制臂：** 在**前向次数**对齐下（DPO 每步 2 次策略前向，故 SFT-on-chosen 拿到约 2 倍优化步数），只在正样本上继续 SFT 达到 0.758，完胜 DPO 的 0.104。所以这里 DPO 的「移动」不是独有价值，更多 SFT-on-positives 反而更好。
- **参考项零结果：** 无 ref（0.141）与带 ref（0.104）在合并标准差内无法区分——KL 锚的稳定作用是大规模/分布漂移下的现象，本微型规模测不出。
- **严重度随 lr 和 1/β 缩放：** 网格 4 格全在基座以下；最温柔的格（lr=3e-4, β=0.3）exact-match 0.204、Δlogp(chosen) 仅 −6.4，最猛的格（lr=1e-3, β=0.1）0.059、−34.9。

## 诚实边界

1. **「偏好」其实是「正确性」：** chosen 是唯一正确答案，没有主观品味维度；这是 **DPO-这个损失** 用在确定性正确性信号 + 易混硬负样本上，**不是**人类偏好、**不是** RLHF。
2. **pref-acc 在基座已近天花板（0.97）：** 排序 chosen>rejected 远比生成正确容易，所以闸门和「目标是否真的在动」用的是**边际可改善性**（14→86，稳健显著），不是会饱和、且受 GPU 非确定性翻转的 pref-acc。
3. **闸门只保证可测，不代表 DPO 好：** `status=pass` 表示对比有效且可测（基座 exact-match 0.588 落在 [0.40,0.85] 头部区间、DPO 损失确实在下降、边际可改善）——而非「DPO 是好的」；本版的诚实结论恰恰是 DPO 让生成回退。
4. **退化对去重：** rejected = 易混 op 在同输入上的输出可能与 chosen 相同（回文的反转、已排序串的排序、`aaa` 上任何 op），这类零梯度对被丢弃并计数（dropped_degenerate=179）。
5. **公平对比：** 带 ref / 无 ref 两臂用**相同基座克隆 + 相同批采样种子**，唯一差别是损失里是否减去 margin_ref；SFT-on-chosen 按前向次数对齐。
6. **规模依赖：** 微型从零 char 级 MiniGPT；病理严重度和参考项零结果都随规模变化。

## 一句话总结

v1166 实现了 DPO 损失并诚实测得它的本性：在一个训弱的 SFT 基座上，DPO 把偏好边际拉高约 6 倍（14→86），但因为它优化的是**相对**边际，logp(chosen) 反而下跌、held-out 生成从 0.59 崩到 0.10——而在前向次数对齐下只做正样本 SFT 反而升到 0.76；参考/KL 项在本规模无可测作用。偏好准确率上升 ≠ 能力提升，这正是这个实验存在的意义。
