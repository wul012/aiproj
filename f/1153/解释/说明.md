# v1153 运行截图和解释

## 目标

v1153 的目标不是再做一次诊断，也不是重新训练，而是把 v1152 的诊断结论真正转化成下一轮可训练的修订语料。v1152 已经把问题收敛得很清楚：v1151 在 target-free holdout prompts 上能稳定命中 `fixed`，但 `loss` 仍然完全缺失，full-pair 还是 0。换句话说，模型已经在局部信号上学到了“先固定”，却没有在后缀上学到“再收尾”。v1153 就是围绕这个缺口构造新的 seed corpus，专门补 `fixed -> loss` 的续写。

## 明确不做什么

本版不做三件事。第一，不训练模型，因此不会把 v1153 解释成能力提升结果。第二，不改动 v1149-v1152 的历史产物，历史版本仍然原样保留，便于复核。第三，不把 target-free holdout prompts 变成带目标词的评估样本；holdout 仍然必须保持 `fixed`/`loss` 词面清洁，否则后续 replay 会失真。

## 输入来源

- v1152 诊断报告：`f/1152/解释/unassisted-holdout-repair-partial-signal-diagnostic-v1152/unassisted_loss_suffix_repair_seed_v1153.json` 的上游依据是 v1152 diagnostic JSON。
- v1149 seed corpus：`f/1149/解释/unassisted-holdout-repair-seed-corpus-v1149/unassisted_holdout_repair_seed_corpus_v1149.json`

## 代码链路

本版核心文件是 `src/minigpt/unassisted_loss_suffix_repair_seed_v1153.py`。它读取两个上游 JSON report 后，做三层处理：

1. 先把 v1149 seed corpus 的原始 rows 变成 `base_rows`。这一步保留原始 full_pair、fixed_first、loss_after_model_fixed 和 short_pair 语义，不回头篡改历史样本。
2. 再根据 v1152 的 `fixed_only_case_ids` 和 `zero_hit_case_ids` 生成 `repair_rows`。`fixed_only_case_ids` 对应新增的 `loss_suffix_after_fixed` 样本，也就是 prompt 末尾手动补上 `fixed`，completion 只训练 `loss`；`zero_hit_case_ids` 对应新增的 `zero_hit_full_pair_reinforcement` 样本，也就是保持 target-free prompt，但直接示范 `fixed loss` 的完整续写。
3. 最后把 base rows 和 repair rows 合并，生成新的 `corpus_text`、JSONL、holdout prompt 文件和 train hint。

这三层处理很关键。它说明 v1153 不是在原有 corpus 上机械复制，而是在局部问题点上增加针对性样本：既补 `loss` 的后缀，也补 `fixed loss` 的完整对照，同时不破坏 target-free 评估面。

## 新增数据结构

### 1. `loss_suffix_after_fixed`

这是本版最核心的新增样本。它的 prompt 形态是 `... fixed`，completion 是 ` loss`。它被标成 `training_only_context=True`，因为 prompt 已经包含了目标词 `fixed`，它不是后续 replay 的 target-free holdout prompt。它的存在目的是告诉模型：在看到 `fixed` 后，`loss` 是一个需要继续发出的后缀，而不是一个独立的起始 token。

### 2. `zero_hit_full_pair_reinforcement`

v1152 里还有一个 zero-hit case：`completion:`。这类样本没有命中 `fixed` 也没有命中 `loss`。v1153 为它补一个 target-free 的 `fixed loss` 完整演示，保证模型不会只在 `answer:` 或 `finish:` 上学会固定模板，而在更难的 `completion:` 上完全失灵。

### 3. `repair_source_case_id`

每一条新增修订样本都记录 `repair_source_case_id`，让后续 review 可以追溯它是从哪个 v1152 case 推导出来的。这个字段很重要，因为它把“修订数据为什么在这里出现”变成可审计事实，而不是口头解释。

## 修订后的 corpus 结构

真实 v1153 运行里，`base_example_count=9`，`repair_example_count=6`，合起来 `revised_example_count=15`。其中 5 条是 `loss_suffix_after_fixed`，1 条是 `zero_hit_full_pair_reinforcement`。`training_only_context_count=6` 说明修订后的新增数据全部都属于训练上下文，不会污染 holdout 测试。`corpus_char_count=347` 相比 v1149 的 198 字符明显变大，但仍然属于 tiny 级别，适合后续用 CPU bounded training 做一次可控验证，而不是上大规模训练。

这个数字变化也说明本版做的是“定向加厚”，不是“盲目扩容”。`base_example_count=9` 还在，历史样本没有被删；新的 6 条修订样本只是把问题最明显的缺口补上。这样下一版训练如果还是没有 `loss`，就能更明确地说明是模型泛化能力不足，而不是语料根本没给足。

## 测试覆盖

v1153 的测试文件覆盖四个关键点。

第一条测试验证修订种子可正常 materialize，并且摘要字段对得上真实结构：base 9、repair 6、revised 15、loss suffix 5、zero-hit full-pair 1、target-free holdout 5。还会断言新增的 `loss_suffix_after_fixed` 行全部是 `training_only_context=True` 且 prompt 中包含 `fixed`，以确保这类样本不会意外变成 holdout 数据。

第二条测试把 v1152 diagnostic 的状态改成 fail，确保 v1153 不会在上游诊断失败时继续产出 seed revision。这种失败会直接把 `status` 变成 fail，说明输入边界被正确拦住。

第三条测试确认 holdout prompts 仍然是 target-free。它检查生成的 5 条 holdout prompt 里没有 `fixed` 或 `loss` 词面泄漏，防止评估面被污染。

第四条测试验证输出和 CLI wiring：目录输入可以被正确解析，`write_unassisted_loss_suffix_repair_seed_v1153_outputs()` 会写出主报告和四类文本/网页输出，同时还会落地 corpus、JSONL、holdout prompts 和 train hint 这几个训练侧 sidecar。

## 下一步角色

v1153 的结尾没有把模型能力往前宣称半步。它只做了两件事：第一，把 v1152 的诊断变成可训练的修订语料；第二，把下一版的动作固定成 `run_unassisted_loss_suffix_repair_training`。也就是说，从 v1153 开始，路线重新回到“基于解释结果修改数据，再进行 bounded training”的闭环。
