# v1123 运行截图与解释

## 本版目标
v1123 把 v1121 的 lookup-only receipt 和 v1122 的 contract check 合并成新的 receipt index。它解决的问题是：下游如果只需要查阅“这份 receipt 是否已经被复核、是否仍然禁止 promotion、是否有稳定 lookup key”，不必分别打开 receipt 与 check 两个目录。

本版不训练模型，不改变 v1121 receipt，不修改 v1122 contract check，也不把 `index_ready=True` 解释成生产发布批准。它只证明 receipt/check 组合可以被只读索引消费。

## 输入与输出
输入：

```text
f/1121/解释/receipt-v1121
f/1122/解释/receipt-check-v1122
```

输出：

```text
f/1123/解释/receipt-index-v1123/*.json
f/1123/解释/receipt-index-v1123/*.csv
f/1123/解释/receipt-index-v1123/*.txt
f/1123/解释/receipt-index-v1123/*.md
f/1123/解释/receipt-index-v1123/*.html
f/1123/图片/v1123-receipt-index.png
```

## 关键运行结果

```text
status=pass
index_ready=True
lookup_scope=downstream_governance_lookup_only
lookup_key_count=1
source_evidence_count=2
lookup_ready=True
contract_check_ready=True
promotion_ready=False
passed_check_count=25
failed_check_count=0
```

这些字段说明 v1123 只建立下游治理查阅索引：receipt 与 contract check 都存在，关键字段一致，lookup key 数量仍然是 1，source evidence 数量仍然是 2，且 promotion 继续保持关闭。

## 截图说明
`f/1123/图片/v1123-receipt-index.png` 是用 Playwright MCP 打开的 v1123 HTML 报告截图。页面中可以看到：

- `Receipt Index Rows`：展示新的 lookup-only 索引行。
- `Source Evidence`：展示 receipt 与 receipt check 两个上游证据。
- `Checks`：展示本版 25 项检查全部通过。

## 一句话总结
v1123 把已复核的 v1121 receipt 与 v1122 contract check 包装成新的 lookup-only receipt index，让下游治理查阅有了稳定入口，同时继续阻止任何 promotion 解释。
