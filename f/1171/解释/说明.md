# v1171 工程保养：抽取脚本单语料初始化（setup_single_corpus），并诚实否决高风险的大规模搬迁

## 背景：用户提了"大保养"，但要"先判断是否合适再做"

到 v1170 已连续 4 个功能版本（v1167 dedup → v1168 → v1169 → v1170），按 AGENTS.md 节奏该做一次保养。用户给了 4 项建议，并明确要求"judge if fine and do"。本版的核心动作之一就是**判断**：先用只读命令核对事实，再决定做什么、否决什么。

## 先核对事实（提案数字基本翻倍夸大）

| 提案说法 | 实测 |
|---|---|
| src/minigpt 有 2467 个文件 | **1226** 个 .py（全部顶层，唯一子目录是 `__pycache__`） |
| 测试 1336 个 | **665** 个 |
| 历史递归命名文件 | `randomized_holdout_publication_*` 确有 **404** 个（最长 205 字符，是 AI 递归拼接 bug 的产物）；`*artifacts*` 524；`bounded_objective_*` 94 |
| model.py 接近/超过 500-800 行 | **346 行**（阈值的 ~43%，远未触发拆分；最大现役模块 dpo_preference_v1166 也只有 731，在区间内） |
| 历史模块耦合 | 抽样 `randomized_holdout_publication_constants` 被 **237 个文件**（202 src + 35 tests）直接 import |

## 判断：只做唯一安全、契约保持的抽取；其余否决

**做（contract-preserving）：** 6 个现役 `scripts/run_*` 入口里，5 个开头是**字节级相同**的单语料初始化（`build_sft_corpus → CharTokenizer.train → pad/eos → block_size=max(16,…)`）。抽进新模块 `src/minigpt/script_setup.py` 的 `setup_single_corpus(...)`，迁移这 5 个脚本。这正是 v1159/v1163/v1167 的节奏：加一个 helper、只迁现役驱动、历史与 v1165（双语料）原样不动。

**否决（实测高风险、违反冻结规则——并非"fine"）：**
- **P1 把 1000+ 历史文件搬到 archive/**：违反"历史长命名/目录只止血/索引/低风险桥接，不做无兼容迁移的改名搬迁"规则；237 个文件 import 历史模块，搬迁会在**收集阶段**就 import 失败——这是破坏性变更而非清理。
- **P2 改名/截断长文件名**：同一规则；改名会切断 git 历史，**抹掉递归命名 bug 的取证痕迹**（`..._receipt_index_receipt_index_..._v1014` 正是 bug 现场），并破坏 237 处 import。
- **P3.2 拆分 model.py**：它只有 346 行，远未到 500-800；过早拆分只会增加跨层 import 耦合。
- **P3.3 重排成 core/training/inference 子包**：要重写 1226 个模块 + 665 个测试里所有 `minigpt.*` import，无兼容迁移=大规模破坏。
- **P4 archive 后跑 --cov**：依赖被否决的 P1；在不完整搬迁后跑覆盖率只会得到假阴性。当前树直接 `pytest` 才是本次保养的放行闸门。
- **模块级抽取**：`_param_l2_delta` 仅 2 处（未达 v1167 的"3+ 才抽"规则）；report 骨架/per-seed 循环是 duck-typed 且 RNG-stream 耦合（每模块派生子种子公式不同），留在原地。

## 为什么契约保持（现有测试不改即绿 = 行为未变的证据）

1. **纯代码搬移**：helper 内逐字复刻那 5 条语句，入参相同。
2. **不碰全局 RNG**：`build_sft_corpus` 内部用自己的 `random.Random(seed)`，`CharTokenizer.train` 是纯 `sorted(set(text))`；helper 不触碰 torch/random 全局流，`seed_everything`/`choose_device` 仍留在各脚本 main() 且顺序不变 → 之后各 seed 的模型初始化权重逐位不变。
3. **不改模块契约**：`src/minigpt/*_v11xx.py` 的 `run_*` 签名未动；单元测试调用的是这些模块函数，根本不经过脚本 main() 的初始化块，观测不到本次重构。
4. **v1166/68/69 的次序微调无副作用**：`build_confusable_preferences` 从"在 build_sft_corpus 与 tokenizer 之间"挪到 helper **之后**——它只消费 `corpus.train/heldout`、与 tokenizer 互不依赖、均不碰 RNG。

## 运行证据

新增 `tests/test_script_setup.py` 三测：**回归守卫**（helper 与旧 inline 序列对 vocab_size/pad_id/eos_id/block_size/`corpus.stats()`/逐样本编码逐一相等）、确定性、block_size 下限与跟随。迁移后 5 个脚本 `py_compile` 通过、无残留死 import（v1169 正确保留仍在用的 `EOS`/`INPUT_ALPHABET`），v1164 端到端 smoke 正常（`block_size=16` 由 helper 给出）。全量 `pytest` 保持绿（仅新增 3 个测试，既有测试不改）。

## 一句话总结

保养的"大"在于**判断的覆盖面**而非搬动的文件数：核对事实后只做了唯一安全的脚本初始化抽取（契约保持、回归守卫），并据冻结规则与 237 文件 import 耦合**诚实否决**了会破坏导入/历史/测试的大规模搬迁与改名。
