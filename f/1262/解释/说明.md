# v1262 运行证据说明

本版完成 production-excellence A1 的第二部分：在 v1261 ruff 增量门之后，引入有限范围、严格模式、可持续扩展的 mypy 类型门。它属于工程/治理线，不训练模型，不修改缓存实验产物，不改动 `decide()` 判定逻辑，也不提升任何模型质量或发布等级。

## 输入

- 范围清单：`docs/static-analysis/mypy-scope.json`
- mypy 配置：`pyproject.toml`
- 八个承重目标：共享报告工具、CI workflow hygiene 三个模块、工程健康两个支撑模块，以及 ruff/mypy 两个分析入口
- CI 顺序：workflow hygiene -> ruff -> mypy -> 其余证据检查 -> coverage

## 输出

`解释/type-analysis/` 保存同一次真实运行生成的 JSON、CSV、Markdown、HTML。关键结果为：

```text
status=pass
decision=continue_with_typed_scope
target_count=8
scope_floor=8
diagnostic_count=0
scope_issue_count=0
```

`图片/type-analysis-v1262.png` 由 Playwright MCP 打开归档 HTML 后截取。截图同时展示状态、决策、目标数、范围下限、诊断数、范围问题数和八个具体文件，因此它证明的不只是“命令返回 0”，还证明报告面向人工复核时可读。

## 解释边界

本版采用 `follow_imports=skip`，含义是严格检查清单中声明的八个文件，而不是顺着导入递归检查全部历史模块。仓库拥有上千个长期演进模块；A1 计划明确禁止一次性全仓机械迁移。该设置把接入成本限制在可审查范围内，同时通过 `scope_floor=8`、目标存在性、唯一性和分组完整性检查阻止范围静默缩小。后续应增加 floor 和目标组，不应降低现有下限。
