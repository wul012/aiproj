## 五版本结果摘要

v1268 将 workflow 限定为 `main` push 与 pull request，抑制 tag 重复执行，并加入 pip cache 与同 ref 并发取消。v1271 同一 commit 同时推送 main 与 tag 后，GitHub Actions 只产生一个 push run，run `29061484360` 最终成功。

v1269 将 staged ruff baseline 从 545 收紧到 271，减少 274 条，并让已有 baseline 的更新只能接受 issue 子集。剩余问题继续可见；本批没有用更新 baseline 的方式接纳新错误。

v1270 将 CI hygiene 的类型、检查构造、摘要推导和公共编排拆开。保留证据记录入口从 523 行到 80 行，固定输入 report 的 63 checks、61 summary keys 和 canonical hash 不变，mypy scope 从 16 增至 19，file-size warning count 从 22 降为 21。

v1271 把九个活跃治理 loader 收敛到严格与宽容两种 JSON 读取契约，累计保护 14 个模块，目标内私有副本为 0；又把格式化后跨过 warning 的 handoff guard 提取为 113 行组件，主模块为 413 个物理行，mypy scope 增至 20，warning count 保持 21。

v1272 没有再建 checker。它运行两种全量发现，检查最终 coverage，并针对报告中唯一的活跃工程线 0% 模块补进程内测试。该模块原本只由 subprocess smoke 间接执行，父 coverage 无法归因；新测试直接覆盖成功聚合、上游状态漂移、selected count 越界、reason scope 失败、缺失 sidecar、malformed scope 与整数回退，最终为 59/63。

## 仍未解决的事实

本版不声称仓库已经“完美”。ruff baseline 仍有 271 条；500 行以上 warning 仍有 21 个，800 行以上的八个历史测试靠显式 no-growth waiver 管理；广义 loader census 仍有 496 个历史文件。coverage 的最低列表还包含多个冻结科学实验模块，但工程线没有权限为了提高数字改写其训练或判断语义。

两种全量验证各需约 17 分钟，这也是实际维护成本。未来若优化测试时长，应先 profile 慢测试、验证 discovery 等价，再考虑分层或并行；不能通过删除历史覆盖、降低 floor 或只跑 smoke 来制造速度收益。

## 停止决定

深度保养在 v1272 停止。继续做同类版本的边际收益已经低于新增版本与文档成本。后续应回到真实需求驱动；只有 static baseline 出现新问题、file-size ratchet 触发、活跃模块职责再次变宽、CI 重复执行回归或生产需求明确出现时，再开启定向保养。
