# v1165 base→SFT 迁移（预训练对下游新指令的数据效率）

## 目标

补上 v1164 明确声明跳过的一步——真正的两阶段 SFT 配方：**先预训练一个基座，再在其上做 SFT**。本版测量这个配方存在的根本原因：在 {复制 C / 反转 R / 排序 S} 上全量预训练一个基座，然后在一个**留出的新指令**——shift-left（`abcd→bcda`，左移一位，与复制/反转共享"定位并拷贝"原语）——上做 completion-only SFT，对比「预训练基座→SFT」与「从零 SFT」在多个 SFT 预算下的表现。

## 运行命令

```powershell
python scripts\run_sft_pretrain_transfer_v1165.py --device cuda --seeds 3
```

## 关键结果（真实 GPU 运行，RTX 4060，3 seeds）

- `status=pass`、`decision=sft_transfer_measured`、`verdict=pretraining_improves_downstream_sft`
- 新指令 shift-left **从未出现在基座预训练里**，所以任何差距都是「共享原语 + 指令格式」的迁移，不是数据泄漏。
- 数据效率曲线（新指令 held-out exact-match vs 下游 SFT 步数）：
  - 50 步：预训练 **0.313** vs 从零 **0.022** → 迁移增益 **+0.291**（约 14 倍）
  - 150 步：0.556 vs 0.285
  - 400 步：0.857 vs 0.488
  - 1000 步：**0.971** vs 0.826 → 增益 +0.146（仍显著）
- 即**预训练把 SFT 曲线整体左移**：预训练基座在 150 步就达到从零在 400 步的水平。这正是"预训练→微调"配方存在的核心原因——下游数据少时迁移收益最大，预算大时两者收敛但预训练仍领先。

## 诚实边界

1. **不是泄漏：** 下游 op（shift-left）不在基座的 {C,R,S} 里；迁移来自共享的定位/拷贝原语与指令格式，不是见过答案。
2. **共享原语是刻意的：** 新 op 故意选了与预训练 op 同族的"位置拷贝"任务，所以本版展示的是迁移的**机制**，不是"预训练对任意下游都迁移"的普适断言。
3. **公平对比：** 同一 (seed, budget) 下两臂用**相同的 SFT 批采样序列**（`torch.manual_seed` 对齐），唯一差异是初始权重（预训练 vs 随机）。
4. **学习性闸门：** 预训练臂在最大预算下达到 0.971 ≥ 0.5 闸门，`task_learned=True`；否则会报 `downstream_op_not_learned` 而非迁移结论。
5. **规模依赖：** 微型从零基座 + 合成 op，迁移幅度随规模变化；这是机制演示。
