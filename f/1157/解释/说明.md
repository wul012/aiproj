# v1157 MiniGPT LoRA held-out generalization eval

## 目标

本版给 v1156 的 LoRA 能力补上一个真正的泛化评估：构造一个更大的、带真实 held-out 切分的模板化语料，并用“基座 / 全参微调 / LoRA”三臂对照，在不相交的 held-out 集上测量损失和下一字符准确率。它要解决 v1156 的遗留问题——语料太小、验证损失被过拟合主导、不是可靠泛化信号。

本版不追求“LoRA 一定赢”，而是先把评估工具本身做对、做可信，再诚实报告 LoRA 的真实表现。

## 运行命令

```powershell
python scripts\run_lora_heldout_eval_v1157.py --device cuda --base-steps 30 --lora-steps 600 --lora-lr 5e-3
```

## 关键结果（真实 GPU 运行，RTX 4060）

- `status=pass`、`decision=heldout_eval_harness_validated`、`harness_valid=True`
- 语料：`train_char_count=3562`、`heldout_char_count=891`、`heldout_token_count=890`（held-out 句子与训练句子组合不相交）
- 基座 held-out loss `2.058854`，acc `0.671910`
- 全参微调 held-out loss `1.228402`（`delta=-0.830452`），acc `0.684270`
- LoRA（全部 16 个 attention+MLP 线性层，`trainable_ratio_percent=7.5051`）held-out loss `2.016049`（`delta=-0.042805`），acc `0.682022`
- `lora_vs_full_capture_ratio=0.0515`，`lora_verdict=lora_partial_gain`

## 核心发现与边界

“评估工具有效”是本版的真实交付：全参微调把 held-out 损失从 2.06 降到 1.23（远超噪声），证明这个 held-out 指标确实在测真实泛化，而不是 v1156 那种被过拟合主导的噪声。这是把 `status=pass` 的含义钉死在“工具可信”，而不是“LoRA 赢了”。

诚实的 LoRA 结论：即使适配了全部 attention+MLP 线性层（7.5% 参数），LoRA 也只拿到了全参微调 held-out 收益的约 5%。原因是结构性的——MiniGPT 的 token embedding 与输出头是 **权重绑定（tied）** 的，LoRA 把它冻结了，而字符级语言模型的大部分可学信号正好落在 embedding/输出头上，LoRA 够不到。这是关于“LoRA 该适配哪些层”的真实工程教训。

## 下一步

v1158 应在 LoRA 真正擅长的场景里验证它：要么做领域自适应（在 A 上预训练、用 LoRA 适配到 B），要么把适配范围扩展到 embedding/输出头。

## 一句话总结

v1157 交付了一个被验证为可信的 held-out 泛化评估工具，并用三臂对照诚实地量化出：在这个权重绑定的微型字符模型上，attention+MLP 的 LoRA 只能拿到全参微调约 5% 的泛化收益——工具做对了，结论也说实话了。
