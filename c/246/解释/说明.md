# v246 registry coverage regression tracking 运行截图说明

## 本版主题

v246 把 release readiness comparison 里的 coverage regression 接入 run registry。

v245 已经能在 readiness comparison 中识别 coverage percent 下降、coverage gap 增大和 coverage panel 变化。v246 继续把这个结果推到 registry 层，让多 run 索引不只看 loss、rubric、pair delta、CI workflow regression，也能看 coverage regression。

## 截图清单

### 01-registry-coverage-tests.png

该截图来自：

```text
python -B -m unittest tests.test_registry tests.test_registry_split -q
```

结果：

```text
Ran 17 tests
OK
```

它证明 registry 的 facade、data、artifact、render split 仍然兼容，并新增覆盖了 `coverage-regressed` 状态、coverage regression count、CSV 字段和 HTML 展示。

### 02-registry-coverage-smoke.png

该截图来自：

```text
python -B scripts/register_runs.py stable coverage-drop --name stable --name coverage-drop
```

关键结果：

```text
release_readiness_comparison_counts={"improved": 1, "coverage-regressed": 1}
test_coverage_regression_count=1
max_abs_test_coverage_percent_delta=7.5
max_abs_test_coverage_gap_delta=3
release_readiness_coverage_regressions=1
```

它证明 CLI 输出和 registry JSON 都能读到 coverage regression，而不是只在 readiness comparison 内部存在。

### 03-registry-coverage-output-check.png

该截图来自对 registry JSON、CSV、HTML 的结构检查。

关键结果：


它证明 coverage regression 已经进入三种消费面：

- 机器消费：`registry.json`
- 表格消费：`registry.csv`
- 人读消费：`registry.html`



它证明 v246 的 registry 接入没有破坏全项目测试集合。

## 证据边界

v246 证明的是 registry 能索引 coverage regression。

它不重新计算 coverage，不改变 coverage gate 阈值，也不把 coverage regression 等同于模型能力退步。coverage regression 仍然只是测试治理和发布证据链的一部分。

## 一句话总结

v246 让 registry 成为 coverage regression 的多 run 索引入口：发布覆盖率退步不再只停留在单个 readiness comparison 报告里，而能进入实验登记、排序、导出和人读 dashboard。
