# v545 运行截图和解释：first-token boost top-k2/t0.8 stability

## 本版目标

v544 只剩 seed `1535` 未恢复。v545 复用已有 `colon_immediate_first_token_boost` corpus，在同一高预算和 `top_k=2, temperature=0.8` 解码配置下运行三 seed，判断“多给 f/l 短前缀样本”是否能修复最后的 first-token gap。

本版不新增代码，只做受控负向实验。

## 运行命令

```powershell
python -B scripts\run_model_capability_required_term_pair_colon_immediate_stability.py --out-dir e\545\解释\model-capability-required-term-pair-first-token-boost-topk2-t080-stability --seeds 535 1535 2535 --corpus-mode colon_immediate_first_token_boost --repeat 420 --bridge-repeat 40 --max-iters 2200 --eval-iters 2 --batch-size 16 --block-size 16 --n-layer 1 --n-head 1 --n-embd 64 --learning-rate 0.02 --max-new-tokens 12 --temperature 0.8 --top-k 2 --force --require-pass

python -B scripts\run_model_capability_required_term_pair_colon_immediate_missed_seed_diagnostic.py e\545\解释\model-capability-required-term-pair-first-token-boost-topk2-t080-stability --out-dir e\545\解释\model-capability-required-term-pair-first-token-boost-topk2-t080-missed-seed-diagnostic --top-k 8 --force --require-pass
```

## 结论

v545 排除了一个看似自然的修法：在当前 tiny char-level 设置中，简单堆叠 `fixed:f`、`loss:l`、`fixed:fi`、`loss:lo` 等短前缀样本没有增强 pair-full，反而让三 seed 全部失稳。后续应保留 v544 的 `colon_immediate + topk2/t0.8` 作为当前最佳基线，再尝试更结构化的对比/校准数据，而不是继续堆短前缀。

一句话总结：v545 是必要的负向实验，证明 first-token boost corpus 不是修复 seed `1535` 的正确方向。
