# v521 运行截图与解释

## 本版主题

v521 对 v518 focus generation rows 做只读 segment audit。v520 已经确认 focused repair 引入了 normalized full signal；本版继续追问：strict miss 到底是由什么输出形态造成的。

边界：本版不训练模型、不修改 tokenizer、不重跑 generation。它只读取 v518 focus 报告，分析 continuation 中 `loss` 字符序列跨过的分隔符，并尝试加载 seed tokenizer 记录 token pieces。

## 运行命令

```powershell
python -B scripts\run_model_capability_required_term_pair_loss_alias_segment_audit.py `
  e\518\解释\model-capability-required-term-pair-loss-alias-focus-metrics `
  --out-dir e\521\解释\model-capability-required-term-pair-loss-alias-segment-audit `
  --force `
  --require-pass
```

## 关键结果

- `status=pass`
- `decision=required_term_pair_loss_alias_newline_segment_boundary`
- `segment_audit_decision=loss_alias_segment_newline_split`
- `case_count=4`
- `strict_miss_normalized_hit_count=4`
- `normalization_gain_count=4`
- `newline_gain_count=4`
- `space_gain_count=0`
- `mixed_gain_count=0`
- `tokenizer_loaded_count=4`
- `dominant_separator_kind=newline`
- `model_quality_claim=tiny_loss_alias_formatting_boundary_observed`

case 级别结果：

| case | strict | normalized | gain | separator | tokenizer |
| --- | --- | --- | --- | --- | --- |
| source-loss | False | True | True | newline | char |
| heldout-beta-loss | False | True | True | newline | char |
| heldout-omega-loss | False | True | True | newline | char |
| heldout-theta-loss | False | True | True | newline | char |

结论：v518 的 normalized full 不是空格或标点导致，而是全部来自 newline segmentation，例如 `los\ns` / `los\nss`。下一步更应该测试 decoding cleanup 或 stop-token/换行处理，而不是立刻改变训练语料。
