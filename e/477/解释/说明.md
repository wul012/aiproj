# v477 运行截图和解释：model capability token budget probe

## 本版目标

v477 接着 v476 的诊断结论做真实验证：既然 v476 发现 `case-token-cap=4` 下大量 prompt 被短输出和 task shape 卡住，本版就用同一个 seed 和同一组 tiny training ladder 对比 `case-token-cap=4` 与 `case-token-cap=12`，观察更长输出是否能减少 prompt-level stall。

本版不扩大模型，不引入外部数据，也不声称模型能力已经提升。它只回答一个更窄的问题：更长输出 budget 是否解除一部分评估阻塞。

## 运行命令

```powershell
python -B scripts\run_model_capability_token_budget_probe.py `
  --out-dir e\477\解释\model-capability-token-budget-probe `
  --token-caps 4,12 `
  --max-iters-list 1,4 `
  --suite-name standard-zh `
  --eval-iters 1 `
  --batch-size 2 `
  --block-size 8 `
  --n-layer 1 `
  --n-head 1 `
  --n-embd 8 `
  --seed 1337 `
  --require-pass `
  --force
```

CLI 输出保存到：

```text
e/477/解释/model-capability-token-budget-probe-cli.txt
```

## 关键结果

- `status=pass`
- `decision=token_budget_probe_ready`
- `summary_decision=longer_token_budget_reduces_eval_stall`
- `baseline_token_cap=4`
- `largest_token_cap=12`
- `token_budget_or_shape_limit_delta=-9.0`
- `persistent_fail_count_delta=-9.0`
- `score_improved_count_delta=0.0`
- `pass_transition_count_delta=0.0`
- `avg_score_delta_change=0.0`

解释：更长 token budget 明显减少了 token/shape stall 和 persistent fail，但还没有带来首尾 ladder 之间的 rubric 分数提升。
