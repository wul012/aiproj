# v527 运行截图和解释

## 本版目标

v527 把 v526 的 core `blocked_token_texts` 能力放到一个新训练 checkpoint 上复验。它不是继续证明旧 v523 checkpoint 的 newline suppression 有效，而是回答一个更实际的问题：当重新训练一个 fresh focus checkpoint 时，strict loss-alias 命中到底来自 blocked-token 解码，还是来自这次训练本身。

本版不把 blocked-token 设置为默认策略，也不改变 playground/API 默认行为。

## 真实运行命令

```powershell
python -B scripts\run_model_capability_required_term_pair_loss_alias_blocked_token_fresh_compare.py `
  e\515\解释\model-capability-required-term-pair-loss-alias-stability `
  --out-dir e\527\解释\model-capability-required-term-pair-loss-alias-blocked-token-fresh-compare `
  --seeds 527 `
  --force `
  --require-pass
```

## 关键结果

```text
status=pass
decision=required_term_pair_loss_alias_blocked_token_fresh_baseline_already_strict
fresh_focus_decision=required_term_pair_loss_alias_focus_support_full_hit
fresh_focus_surface_decision=loss_alias_focus_support_full_hit
blocked_token_surface_decision=loss_alias_newline_suppression_baseline_already_full
case_count=4
baseline_strict_hit_count=4
blocked_token_strict_hit_count=4
blocked_token_strict_gain_count=0
```

解释：seed `527` 的 fresh focus checkpoint 在默认解码下已经实现 `4/4` strict hit，因此 v527 不能被解释成 blocked-token 带来的新增恢复。它更像是一个“fresh seed 本身更强”的证据，说明下一步应比较多个 fresh seeds，而不是马上把 blocked-token 当成默认解码策略。
