# v702 运行截图和解释：minimal prompt balanced training

## 本版目标

v702 按 v701 的 balanced repair plan 做一次真实 tiny training。它使用同一个 seed `3535`，保持 v696/v699 的训练规模，只把 corpus mode 改成 `minimal_prompt_balanced_first_token_repair_objective`。

本版目标不是证明模型变好，而是验证 balanced first-token repair 是否能让 minimal prompt 同时命中 `fixed` 和 `loss`。

## 运行命令和结果

```powershell
python -B scripts\run_model_capability_required_term_pair_coexistence_refresh.py --out-dir e\702\解释\model-capability-required-term-pair-minimal-prompt-balanced-training --seed 3535 --corpus-mode minimal_prompt_balanced_first_token_repair_objective --repeat 260 --bridge-repeat 20 --max-iters 1400 --eval-iters 2 --batch-size 16 --block-size 16 --n-layer 1 --n-head 1 --n-embd 64 --learning-rate 0.02 --max-new-tokens 12 --temperature 0.2 --top-k 1 --device cpu --require-pass --force
```

结果：

```text
status=pass
decision=required_term_pair_coexistence_refresh_no_pair_full
training_status=pass
checkpoint_exists=True
pair_full_observed=False
model_quality_claim=not_claimed
```

生成样本显示：

```text
fixed= -> fixed=lix=los=loss
loss=  -> loss=lossssss=los
```

这说明 balanced corpus 仍没有让 tiny checkpoint 学会双目标共存；它把 `fixed=` 分支污染到 loss-like 模式，`loss=` 分支则继续命中 loss。

## 结论

v702 是一个必要的负结果：

- checkpoint 真实生成，训练链路没有失败。
- balanced corpus 没有带来 pair-full。
- 失败不再只是 fixed-only 或 loss-only，而是固定分支被 loss 字符模式吸附。
- 下一步应避免继续单纯增加同类 minimal prompt 行，转向结构化 pair-readiness 或更明确的 evaluation/training split。

一句话总结：v702 证明 balanced first-token repair 还不足以提升 tiny 模型的 fixed/loss 双目标能力。
