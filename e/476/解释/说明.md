# v476 运行截图和解释：model capability stall diagnostic

## 本版目标

v476 复用 v475 已归档的多 seed ladder 稳定性报告，继续追问一个更具体的问题：为什么两个 seed 都出现 validation loss 下降，但 benchmark score 和 generation flags 仍然没有变化。

本版不重新训练模型，不扩大模型规模，也不把诊断结果包装成模型能力提升。它只做首尾 rung 的 prompt 级别对齐，解释当前 tiny ladder 的评估停滞。

## 运行命令

```powershell
python -B scripts\diagnose_model_capability_stall.py `
  e\475\解释\model-capability-ladder-stability `
  --out-dir e\476\解释\model-capability-stall-diagnostic `
  --require-pass `
  --force
```

CLI 输出已保存到：

```text
e/476/解释/model-capability-stall-diagnostic-cli.txt
```

## 关键结果

- `status=pass`
- `decision=capability_stall_diagnostic_ready`
- `case_count=20`
- `score_improved_count=0`
- `score_degraded_count=0`
- `score_unchanged_count=20`
- `persistent_fail_count=20`
- `token_budget_or_shape_limit_count=20`
- `avg_score_delta=0.0`
- `summary_decision=token_budget_or_shape_limits_block_eval_signal`

这说明 v475 的 loss 改善没有传导到 case 级别的 rubric 分数。当前最主要的阻塞不是“seed 不稳定”，而是 tiny smoke 的输出 token 太短，导致 `length_bounds`、`task_shape` 和 `must_include` 持续失败。
