# v484 运行截图和解释：required-term holdout

## 本版主题

v484 承接 v483 的结果：全量 scaffold-to-term 微训练后，20 个 prompt 中出现了 4 个 continuation required-term hit。本版继续追问：这些命中是否能跨过训练样本本身，还是只在同批样本上偶然出现？

本版做 held-out split：训练语料只包含部分 required terms 的 scaffold-to-term 配对；另一部分 required terms 只暴露 prompt/vocabulary，不写入 `prompt -> term` 重复样本。然后同时评估 train split 和 holdout split 的 continuation 命中。

```text

python -B scripts/run_model_capability_required_term_holdout.py e/483/解释/model-capability-required-term-micro-training --out-dir e/484/解释/model-capability-required-term-holdout --max-iters 420 --eval-iters 2 --batch-size 16 --block-size 8 --n-layer 1 --n-head 1 --n-embd 32 --learning-rate 0.02 --term-repeat 24 --max-new-tokens 8 --temperature 0.2 --top-k 1 --generation-seed 484 --device cpu --require-pass --force
```

## split 设计

默认按 term 做 deterministic split，避免同一个 required term 同时进入训练和 holdout。

```text
train_terms=because, chain, fixed, four, real, text
holdout_terms=data, loss, while
train_example_count=12
holdout_example_count=8
```

语料里会保留 holdout prompt vocabulary，例如 `data:`，这样 char tokenizer 可以编码 prompt；但不会写入 `data:data` 这样的 holdout prompt-to-term 训练样本。

## 关键结果

```text
status=pass
decision=required_term_holdout_no_uptake
holdout_decision=heldout_micro_training_no_required_term_uptake
train_example_count=12
holdout_example_count=8
train_continuation_hit_count=0
holdout_continuation_hit_count=0
train_hit_rate=0.0
holdout_hit_rate=0.0
training_status=pass
checkpoint_exists=True
model_quality_claim=not_claimed
```

解释：v484 的 stricter split 下，训练侧和 holdout 侧都没有 required-term continuation hit。这说明 v483 的 4/20 命中还不能被解释为稳定学习到 required-term 续写能力；下一步要先在 split 条件下复现训练侧命中，再谈 held-out 泛化或扩大 benchmark。

## 结论

v484 把 v483 的定向训练信号放进更严格的 split 里复测，结果没有复现 continuation required-term uptake。项目应该先改善微训练语料与 split 设计，让训练侧信号稳定，再继续讨论 held-out 泛化和模型能力提升。
