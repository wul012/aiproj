# v499 运行截图和解释

## 本版主题

v499 接在 v498 后面。v498 已经证明：只改变生成长度、temperature 和 top-k，不能让 `fixed/loss` pair 从 partial hit 恢复到 full hit。v499 因此不继续训练，也不继续扫解码参数，而是反查 v497 的 capacity corpus，检查训练行本身有没有把两个目标词混在同一个 prompt 后面。

本版不声明模型质量提升。它只回答一个更基础的问题：`fixed:` 和 `loss:` 这些 prompt 在训练语料里是否真的保持了目标分离。

## 真实运行命令

```text
python -B scripts/run_model_capability_required_term_pair_prompt_separation_audit.py e/498/解释/model-capability-required-term-pair-decoding-sweep --out-dir e/499/解释/model-capability-required-term-pair-prompt-separation-audit --force --require-pass
```

输入源：

```text
e/498/解释/model-capability-required-term-pair-decoding-sweep/model_capability_required_term_pair_decoding_sweep.json
```

它会继续反查：

```text
e/497/解释/model-capability-required-term-pair-capacity-sweep/model_capability_required_term_pair_capacity_sweep.json
```

## 关键结果

```text

target_count=2

corpus_revision_recommended=True
model_quality_claim=not_claimed
```

真实观察：

- 两个 v497 partial target 的 capacity corpus 都能读到。
- 每个 target 有 `fixed` 和 `loss` 两个 term，一共 4 个 term row。
- 每个 term row 都有 240 行类似 `fixed:fixed not loss` 或 `loss:loss not fixed` 的训练行。
- 这些行让“另一个 required term”直接出现在同一个 prompt 后的 continuation 位置。
- 所以 v497/v498 的 partial 行为更像 corpus row design 问题，而不是单纯训练步数或解码参数问题。

v499 把 v498 的“不是解码能解决”进一步落到语料证据：当前 pair corpus 把 contrast 负例写在同一 prompt 后面，下一步应先做 contrast-free pair corpus，再训练新的小模型。
