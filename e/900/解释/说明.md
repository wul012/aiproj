# v900 运行截图与解释

## 本版目标

v900 接在 v899 后面，执行 tokenizer-coverage-aware holdout real replay。它使用 v898 suite、v899 dry-run 和 v890 checkpoint/tokenizer，检查排除中文 tokenizer 覆盖问题后，模型是否能在 5 条 ASCII holdout prompt 的 continuation 中命中 `fixed/loss`。

- Holdout suite: `e/898/解释/bounded-objective-loss-signal-bridge-target-only-memory-tokenizer-coverage-aware-holdout-suite`
- Dry-run: `e/899/解释/bounded-objective-loss-signal-bridge-target-only-memory-tokenizer-coverage-aware-holdout-dry-run`

## 解释

v900 是这一段路线里第一条真正正向的 holdout replay 证据。v896 的 unchanged 中文 holdout 只有 1/5，通过 v897 发现主因是 tokenizer coverage gap；v898 生成 tokenizer-covered suite；v899 证明 scoring contract 可用；v900 用同一个 v890 checkpoint 真实 replay 后，5 条 case 的 continuation 都命中 `fixed/loss`。

这仍然不是自动 promotion，因为 v898 suite 是新构建的 tokenizer-aware holdout，需要 review 它是否足够独立、是否会因为 prompt 中包含 `fixed loss` 而过于容易。但它确实说明：当 prompt 可以被 tokenizer 表达时，v890 checkpoint 已能在这些 bounded objective prompt 上生成目标 terms。

v900 证明 v890 checkpoint 能通过 tokenizer-covered holdout suite，但 promotion 仍需 review suite 独立性和难度。
