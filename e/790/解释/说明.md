# v790 运行截图和解释：objective-level contrast promotion manifest

## 本版目标

v790 是功能版本，接在 v789 objective-level contrast acceptance review 后面。v789 已经把 v772 的三 seed rollup 接受为 `objective_level_contrast` 路线，v790 继续把这个 accepted review 转成 promotion manifest，让后续 benchmark history 或 portfolio 可以消费一个稳定、可索引的 route promotion 条目。

本版明确不做：

- 不重新训练 checkpoint。
- 不重新运行 pair-probe replay。
- 不扩大模型质量声明。
- 不把 tiny pair-probe 证据解释成生产模型能力。

## 运行命令和结果

```powershell
python -B scripts\run_model_capability_required_term_pair_readiness_objective_level_contrast_promotion_manifest.py --acceptance-review e\789\解释\model-capability-required-term-pair-readiness-objective-level-contrast-acceptance-review --out-dir e\790\解释\model-capability-required-term-pair-readiness-objective-level-contrast-promotion-manifest --require-pass --force
```

输出摘要：

```text
status=pass

failed_count=0
promotion_manifest_ready=True
promoted_route_id=objective_level_contrast

model_quality_claim=seed_stable_pair_probe_route_accepted
```

## 关键观察

- v790 只接受 v789 的 `pair_readiness_objective_level_contrast_acceptance_review_accepted` 决策。
- manifest 中的 `route_status=promoted` 表示路线进入可索引 promotion artifact，不表示模型进入生产级质量。
- `model_quality_boundary` 继续保持 `tiny_required_term_pair_probe_only`。
- `benchmark_history_entry` 已包含 `route_id`、`status`、`boundary`、`model_quality_claim`、`seed_count`、`min_pair_full_count` 和 `pair_full_strength_spread`。
- 测试额外覆盖了 route id 不匹配和 boundary 被改宽两种失败路径。
