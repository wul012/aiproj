# v506 运行截图与解释

## 本版主题

v506 增加 `model_capability_required_term_pair_decoding_path_trace`。它读取 v505 decoding-gap probe，复用同一 checkpoint、profile 和 seed，逐步重放采样路径，记录 expected first token 的 rank、实际第一步采样 token、expected token 首次出现位置，以及最终 continuation。

本版不训练新模型，也不扩大 profile 搜索。它只解释 v505 的局部命中为什么没有变成稳定 pair-level generation。

## 运行命令

```powershell
python -B scripts\run_model_capability_required_term_pair_decoding_path_trace.py `
  e\505\解释\model-capability-required-term-pair-decoding-gap-probe `
  --out-dir e\506\解释\model-capability-required-term-pair-decoding-path-trace `
  --force `
  --require-pass
```

## 关键结果

- `status=pass`
- `decision=required_term_pair_decoding_path_late_expression`
- `decoding_path_trace_decision=decoding_path_trace_late_expression_after_first_miss`
- `probe_count=6`
- `trace_step_count=120`
- `continuation_hit_count=2`
- `first_sample_match_count=0`
- `late_hit_after_first_miss_count=2`
- `first_expected_rank_le_5_count=6`

所有 probe 的 expected first token 排名都不差，但第一步采样没有一次直接采到 expected first token。v505 的两个命中都是 late hit：

- `greedy-24/fixed` 在第 5 步后才出现 expected first token。
- `top2-24/loss` 在第 8 步后才出现 expected first token。

这说明下一步更应该做 first-token constrained repair，而不是继续扩大训练语料。

## 截图说明

`01-model-capability-required-term-pair-decoding-path-trace.png` 展示 v506 HTML 报告。截图重点看三处：

- 顶部 summary 显示 first matches 为 `0`、late hits 为 `2`。

## 一句话总结

v506 把 v505 的局部生成命中解释为“第一步采样错过后晚出现”，为 v507 的 first-token constrained repair 提供了明确目标。
