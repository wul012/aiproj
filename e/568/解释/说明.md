# v568 required-term pair first-token route decision 运行说明

## 本版目标

v567 已经证明 first-token hint 密度调整只是覆盖迁移，不是稳定提升。v568 把这个判断固化成一个轻量 route decision artifact：输入 v567 comparison JSON，输出可机器读取、可截图审阅的路线选择。

本版不训练模型、不新增 corpus mode、不扩大报告链。它只把“停止 first-token density 路线”的判断从 README 文字升级为可复核产物。

## 代码改动

- `src/minigpt/model_capability_required_term_pair_first_token_route_decision.py`
  - 新增 route decision builder。
  - 读取 v567 comparison 的 source reports。
  - 选择 pair-full 数最高且最简单的非 first-token-density route。
  - 标记 full/light first-token route 的拒绝原因。
- `src/minigpt/model_capability_required_term_pair_first_token_route_decision_artifacts.py`
  - 输出 JSON、CSV、text、Markdown、HTML。
- `scripts/run_model_capability_required_term_pair_first_token_route_decision.py`
  - 新增 CLI，可接受 comparison JSON 或输出目录。
- `tests/test_model_capability_required_term_pair_first_token_route_decision.py`
  - 覆盖正常选择、失败输入、输出格式和目录定位。


## 关键结果

```text
status=pass
decision=stop_first_token_density_and_replay_best_baseline
source_report_count=3
first_token_density_route_count=2
stable_route_count=0
max_pair_full_seed_count=1
selected_source_label=v562-loss-balanced
model_quality_claim=route_decision_only
```

route decision 的含义：

- 选择 `v562-loss-balanced`
  - 它和 v564 一样是 `1/3` pair-full，但没有额外 first-token density 复杂度。
- 拒绝 `v564-full-first-token`
  - 原因是 `first_token_density_no_stable_gain` 和 `not_stable_across_seeds`。
- 拒绝 `v566-light-first-token`
  - 原因是 `no_pair_full_seed`、`first_token_density_no_stable_gain` 和 `not_stable_across_seeds`。

## 结论

v568 把 v559-v567 的探索收口为一条清晰路线：停止继续调 first-token hint 密度，先对 `v562-loss-balanced` 做 held-out equals-surface replay，再决定是否需要新 objective。
