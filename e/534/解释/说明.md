# v534 运行截图和解释：required-term pair first-token preference

## 本版目标

v534 读取 v533 的真实 refresh checkpoint，不再训练，只检查 `fixed:` 和 `loss:` 下一 token 的 logits 排名。目标是解释为什么 v533 明明训练成功，却生成 `ansssssssss` 而没有进入 fixed/loss。


## 结论

v534 把 v533 的失败模式定位到第一 token：模型不是先选 `f` 或 `l`，也不是直接先选 `a`，而是极高概率先选空格。`ans...` 是空格之后的后续坍缩。下一版更合理的方向是移除带前导空格的 answer-shaped 行，或增加 `fixed:fixed` / `loss:loss` 这种 colon-immediate 训练行。

一句话总结：v534 把 fixed/loss pair refresh 的问题从“训练没有命中”细化为“第一 token 被空格格式偏置压住”。
