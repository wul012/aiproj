# v943 运行截图与解释

## 本版目标

v943 新增 `randomized_holdout_publication_registry_downstream_consumer_index`，把 v941 的 downstream consumer packet 与 v942 的 contract check 合并成一个只读消费索引。

这版解决的问题是：下游消费者不应该直接散读 packet 和 check 两份产物，而应该拿到一份同时说明“可查什么、不能做什么、源证据是否通过”的索引。它不扩展模型能力声称，也不解除 promotion 阻断。

## 输入与输出

- 输入 1：`e/941/解释/randomized-holdout-publication-registry-downstream-consumer-packet/randomized_holdout_publication_registry_downstream_consumer_packet.json`
- 输入 2：`e/942/解释/randomized-holdout-publication-registry-downstream-consumer-packet-check/randomized_holdout_publication_registry_downstream_consumer_packet_check.json`
- 输出目录：`e/943/解释/randomized-holdout-publication-registry-downstream-consumer-index`
- 截图目录：`e/943/图片`

生成命令：

```powershell
python scripts\build_randomized_holdout_publication_registry_downstream_consumer_index.py --consumer-packet e\941\解释\randomized-holdout-publication-registry-downstream-consumer-packet --consumer-packet-check e\942\解释\randomized-holdout-publication-registry-downstream-consumer-packet-check --out-dir e\943\解释\randomized-holdout-publication-registry-downstream-consumer-index --require-index-ready --require-lookup-ready --force
```

## 边界说明

v943 的索引只允许 governance lookup。它把 v941 packet rows 暴露为 lookup rows，但保留 v942 contract check 的通过状态；如果 packet 被篡改、check 失败、lookup key 漂移、blocked uses 不完整，索引会失败。

一句话总结：v943 把 downstream consumer 的两份源证据合成了一个可检索但不可促生产的只读索引。
