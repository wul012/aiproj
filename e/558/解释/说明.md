# v558 required-term pair constrained decode feasibility 运行说明

## 本版目标

v557 证明 v556 checkpoint 在 `fixed=` 后 teacher-forced 更偏向 `loss`。v558 进一步测试一个 decode-only 补救：如果在 `fixed=` 下阻断竞争项 `loss` 的首字母 `l`，是否能不用重训就恢复 `fixed`。

本版不训练、不改 checkpoint、不改变已有 generation profile；它只新增一个只读 feasibility 诊断。

## 关键改动

- `src/minigpt/model_capability_required_term_pair_constrained_decode_feasibility.py`
  - 读取 refresh report、checkpoint 和 tokenizer。
  - 运行 default 与 `block_competing_initial` 两个 profile。
  - 对 `fixed=` 阻断 `l`，对 `loss=` 阻断 `f`，观察 continuation hit 是否改善。
- `src/minigpt/model_capability_required_term_pair_constrained_decode_feasibility_artifacts.py`
  - 输出 JSON、CSV、text、Markdown 和 HTML。
- `scripts/run_model_capability_required_term_pair_constrained_decode_feasibility.py`
  - 提供 CLI 入口。
- `tests/test_model_capability_required_term_pair_constrained_decode_feasibility.py`
  - 覆盖 pair-full feasible、no gain、无效输入、locator 和 artifacts。

## 核心结果

```text
status=pass
decision=required_term_pair_constrained_decode_not_feasible
default_hit_count=1
constrained_hit_count=0
hit_delta=-1
constrained_pair_full=False
fixed_constrained_hit=False
```

case 级观察：

```text
default fixed -> miss, continuation=los paross f
default loss  -> hit,  continuation=loss fixes p
block l fixed -> miss, continuation=01\npt=01 1 f
block f loss  -> miss, continuation=01\npaross lo
```

阻断竞争首字母没有恢复 `fixed=`，还破坏了原本命中的 `loss=`。这说明当前 checkpoint 不适合用简单 blocked-token profile 补救。
