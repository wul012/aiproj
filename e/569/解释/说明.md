# v569 required-term pair route held-out replay 运行说明

## 本版目标

v568 选择了 `v562-loss-balanced` 作为停止 first-token density 后的保守 baseline。v569 把这个路线选择接到真实 held-out prompt replay：只读取 v568 route decision，找到被选中 v562 报告里的 pair-full seed，再用已有 generation profile replay 在 held-out prompt surfaces 上复核。

本版不重新训练、不新增 corpus mode，也不把单 seed 结果包装成稳定模型能力。它验证的是：v562 里已经 pair-full 的 seed `1535`，离开源 prompt 后是否仍能在 held-out prompt 上保持 fixed/loss 同时出现。

## 代码改动

- `src/minigpt/model_capability_required_term_pair_route_heldout_replay.py`
  - 新增 route-heldout replay builder。
  - 从 route decision 的 `selected_route.source_path` 读取源稳定性报告。
  - 只挑选 `pair_full_observed=True` 的 seed report。
  - 对 `colon-spaced`、`equals`、`arrow` 三种 prompt spec 调用 generation profile replay。
- `src/minigpt/model_capability_required_term_pair_route_heldout_replay_artifacts.py`
  - 输出 JSON、CSV、text、Markdown、HTML，并写出 replay sidecar。
- `scripts/run_model_capability_required_term_pair_route_heldout_replay.py`
  - 新增 CLI。
- `tests/test_model_capability_required_term_pair_route_heldout_replay.py`
  - 覆盖 ready、not-ready、失败输入、输出 sidecar、目录定位。


## 关键结果

```text
status=pass
decision=required_term_pair_route_heldout_replay_ready
row_count=3
heldout_pair_full_count=3
heldout_pair_full_rate=1.0
heldout_all_pair_full=True
model_quality_claim=targeted_route_heldout_ready
```

三条 held-out prompt surface 都通过：

```text
colon-spaced -> pair-full
equals       -> pair-full
arrow        -> pair-full
```

这说明 v562 seed `1535` 的 pair-full 信号不只绑定在 `fixed=` / `loss=` 单一源 prompt 上，但仍然只是一个 seed 的 targeted route held-out 结果。
