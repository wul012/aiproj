# v897 运行截图与解释

## 本版目标

v897 接在 v896 后面，专门解释为什么 v895 已经恢复 `fixed loss` contract 后，v803 unchanged holdout suite 仍只有 1/5 通过。

本版不重新训练，不改 v803 suite，不重新生成 replay。它只读取 v896 holdout replay、v890 tokenizer 和 v890 prepared corpus，量化 tokenizer/corpus/prompt 三类覆盖问题。

## 解释

v890 的 tokenizer 是 49 个字符的 char tokenizer，词表里没有中文字符。v803 的 holdout prompts 主要是中文任务描述，因此 replay 时大量 prompt 字符会被编码为 `<unk>`，解码后表现为 `�` replacement 字符。

v897 进一步确认：

- 5 条 holdout prompt 都包含 unknown 字符。
- 4 条失败 case 被直接归类为 `tokenizer_prompt_coverage_gap`。
- 唯一通过的 `objective-answer-check` 也有 unknown 字符，只是仍碰巧输出了 `fixed/loss`。
- v890 prepared corpus 里包含 `fixed loss`，但不包含这些中文 holdout prompt surface。

因此 v896 的失败不能简单理解为“模型还差一点训练”；它首先是 tokenizer/suite 覆盖不对齐的问题。下一步应该先构建 tokenizer-coverage-aware holdout suite，或重新训练覆盖中文 prompt 的 tokenizer/corpus，再谈更强模型能力。

## 一句话总结

v897 证明 v896 holdout 失败的主因是 tokenizer prompt coverage gap，而不是单纯 decoder budget 或 scoring 问题。
