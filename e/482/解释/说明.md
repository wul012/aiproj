# v482 运行截图和解释：required-term scaffold probe

## 本版主题

v482 承接 v481 的结论：required terms 在 expected behavior 中出现了 212 次，但 archived tiny generation 没有一次把它们写进 continuation。本版做一个真实生成 probe：加载 v478 的 cap-12 `max-iters-4` checkpoint，用极短 scaffold prompt 重新生成，检查显式给出一个 required term 后，模型是否能在 continuation 里继续生成它。

本版不重训模型，不扩大 benchmark，只测试 prompt scaffold 对当前 tiny checkpoint 的即时影响。

## 运行输入

```text
source=e/481/解释/model-capability-required-term-uptake
checkpoints=e/478/解释/model-capability-token-budget-stability/seeds/*/token-cap-12/ladder/rungs/max-iters-4/run/checkpoint.pt
tokenizers=e/478/解释/model-capability-token-budget-stability/seeds/*/token-cap-12/ladder/rungs/max-iters-4/run/tokenizer.json
```

CLI：

```text
python -B scripts/run_model_capability_required_term_scaffold_probe.py e/481/解释/model-capability-required-term-uptake --out-dir e/482/解释/model-capability-required-term-scaffold-probe --max-new-tokens 24 --temperature 0.7 --top-k 30 --generation-seed 482 --term-limit 1 --device cpu --require-pass --force
```

解释：v478 tiny checkpoint 的 `block_size=8` 很小，所以本版只给每个 case 一个最短 required term，prompt 形如 `data:` 或 `text:`。这些 prompt 没有被截断，也没有超过 block size；`scaffold_generated_hit_count=20` 只是说明 required term 出现在 prompt 本身。真正关键的是 `scaffold_continuation_hit_count=0`，说明模型看到短 scaffold 后，仍没有在后续生成里复现 required term。

## 结论

v482 证明：即使用不截断的极短 required-term scaffold，当前 tiny checkpoint 也没有把 required term 续写出来。下一步更适合做 targeted micro-training repeat，而不是继续只调 prompt 或扩大 benchmark。
