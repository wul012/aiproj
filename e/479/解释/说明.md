# v479 运行截图和解释：model capability rubric signal audit

## 本版目标

v479 接着 v478 的结论继续缩小问题范围。v478 已证明 `case-token-cap=12` 在两个 seed 下稳定释放 token/shape 阻塞，但 score/pass 仍然不变。本版不再重跑训练，而是读取 v478 的 cap-12 stall diagnostic，审计剩余失败到底集中在哪些 rubric/data 信号上。

本版不扩大模型，不改 benchmark，不声称模型能力提升。它只回答：cap 12 之后，阻塞 score/pass 的主要证据是什么。

## 运行命令

```powershell
python -B scripts\audit_model_capability_rubric_signal.py `
  e\478\解释\model-capability-token-budget-stability `
  --out-dir e\479\解释\model-capability-rubric-signal-audit `
  --target-token-cap 12 `
  --require-pass `
  --force
```

## 关键结果

- `status=pass`
- `decision=rubric_signal_audit_ready`
- `audit_decision=rubric_required_terms_dominate_flat_scores`
- `target_token_cap=12`
- `seed_count=2`
- `case_count=20`
- `score_improved_count=0`
- `pass_transition_count=0`
- `persistent_fail_count=2`
- `preview_changed_count=8`
- `top_failed_checks=must_include:20, task_shape:2`
- `top_stall_reasons=generation_unchanged:11, required_terms_missing:5, case_passed:2, token_budget_or_shape_limit:2`
- `cross_seed_failed_checks=must_include,task_shape`

解释：cap 12 已经大幅缓解 token budget 问题，但剩余 flat score 主要被 `must_include` required-term 检查支配。下一步更应该检查 tiny corpus 是否覆盖这些 required terms，以及 rubric 是否适合 tiny 模型，而不是直接加大模型。
