# v303 training-governance documentation stage split

## 本版目标和边界

v303 解决的是文档类目录继续堆积的问题。`代码讲解记录_项目成熟度阶段/` 已经承接 v48-v302 的成熟度、训练规模、自动化交接和 CI 回归治理讲解，文件数量明显变多，继续写入会降低检索效率。

本版不改 MiniGPT 训练、评估、CI workflow、报告生成逻辑，也不迁移旧讲解文件。它只把“后续讲解写到同级新目录”这件事落成仓库规则、README 索引和版本证据。

## 前置能力

前置规则来自 AGENTS 的文档分流要求：当文档目录内容明显变多、主题开始分叉、或继续写入会降低检索效率时，可以新建同级目录承接后续内容。v303 是这个规则在 aiproj 的第一次实装。

## 关键文件

- `AGENTS.md`
  - 明确 `代码讲解记录_项目成熟度阶段/` 保留 v48-v302。
  - 明确 v303 起使用 `代码讲解记录_训练治理阶段/`。
  - 保留“不迁移旧文件，只补索引和分流说明”的边界。

- `代码讲解记录_项目成熟度阶段/README.md`
  - 在旧阶段 README 顶部说明该目录已经成为 v48-v302 历史阶段。
  - 指向新的同级目录，避免后续读者误以为还要继续往这里追加。

- `代码讲解记录_训练治理阶段/README.md`
  - 新阶段入口。
  - 写清编号从 `317-v303-...` 继续，不重置历史编号。
  - 说明如果这个目录未来也变密，可以继续开同级阶段目录。

- `README.md`
  - 把当前版本更新为 v303。
  - 在 checkpoint、tag 和 code explanation records 中加入新目录。
  - 用 README 作为入口分流，而不是回头搬迁旧内容。

- `d/303/解释/说明.md`
  - 记录本版运行证据说明。
  - 说明本版是文档治理版本，没有改变训练或报告行为。

## 输入输出和流程

输入是当前仓库的文档结构：

```text
代码讲解记录_项目成熟度阶段/
```

该目录已经保存 v48-v302 的历史讲解。v303 的输出是新的同级目录：

```text
代码讲解记录_训练治理阶段/
```

后续讲解从这里继续写入，例如：

```text
317-v303-training-governance-doc-stage-split.md
```

这个输出不是运行时 artifact，也不是训练结果；它是仓库级文档治理约定，供后续版本持续消费。

## 测试和验证

本版的验证重点不是模型行为，而是结构一致性：

- README 必须引用 `代码讲解记录_训练治理阶段/README.md`。
- AGENTS 必须写明 v303 起新目录承接后续讲解。
- 旧阶段 README 必须说明 v48-v302 已收口。
- `d/303` 必须保存本版运行截图和说明。
- Source encoding 检查必须继续通过，避免文档或源码引入编码异常。

这些断言保护的是“新旧目录边界清楚、后续不会继续堆错目录”。

## 一句话总结

v303 把 aiproj 的文档类增长从单目录堆积改成同级阶段承接，让后续训练治理讲解有了新的、可持续的入口。
