# 1249 - v1292 F401 审计清扫:152 处未用导入,90 保留 62 删除,基线 153 → 1

## 本版目标与不做什么

优雅计划第三级:处理静态基线里最后的大块——152 处 F401("导入但未使用"),
分布在 70 个文件。这类债**绝对不能无脑修**:一个"未使用"的导入可能是被
下游 `from X import Y` 消费的有意 re-export,盲删会断真实的导入链。本版的
全部价值在审计方法:先证明每一条是死是活,再动手。不做什么:不碰唯一剩下
的 F821(data_prep 的潜在 NameError,bug 类,v1293 单独修)。

## 审计三次迭代:两次被现实纠正(如实披露)

**v1 (朴素)**:对每个 (模块, 符号) 搜 `from …模块 import …符号`。
结论 146 死 / 6 活。**错**。

**v2 (加属性访问)**:在动手前抽查 server.py 时发现
`test_foundation_package_reexports.py` 用 `minigpt.server.parse_generation_request`
**属性访问**断言公共 re-export 契约——v1 的正则根本看不见这类消费者。
加上属性模式后:15 活 / 137 死。执行到一半又被测试当场打脸:
`minigpt/reports/manifest.py` 从 `minigpt.manifest` 导入 `utc_now` 用的是
**带括号的多行 import**,v2 的 `[^\n]*` 在换行处断了。回滚。

**v3 (终版)**:三种消费形态全覆盖——单行 from-import、括号多行
from-import(re.S 匹配括号组)、属性访问;并且用**本地名**审计
(`import X as Y` 被 ruff 标记的是别名 Y,不是原名 X)。
结论:**90 活 / 62 死**。从 146 到 62——朴素审计会错杀 84 条真实导入链。
这是本版最重要的教训:**"未使用"是 ruff 的文件内视角,不是仓库视角**;
model_capability 家族的 `read_json_report`/`resolve_exit_code`/FILENAME
常量链是几十个模块间有意的转发协议。

## 执行:原子管线

第一次执行还犯了一个管线错误:formalize 脚本在一个已别名的导入上断言
失败而中途退出,但后续的 `ruff --fix` 没有被 `&&` 守卫、照跑不误,把
还没保护的 keeper 一并删了。回滚,重写为**原子管线**:所有编辑先在内存
里算完、全部断言通过才落盘;`ruff --fix` 只在前一步成功后执行。

- 90 条活导入:89 条改写成 `Y as Y`(ruff 认可的显式 re-export 惯用法,
  且保持 `is` 同一性——公共 API 契约测试原样通过);1 条已有不同别名的
  改加 `# noqa: F401 (re-export)`。
- 62 条死导入:`ruff --fix` 自动删除。
- 结果:F401 全仓库归零。

## 工程教训

1. **删除类清理的审计必须覆盖全部消费形态**:from-import 单行、多行
   括号组、属性访问、别名。少一种形态就多一批错杀。真正救场的是契约
   测试——公共 API 的 `is` 断言把 v2 的错删当场暴露。
2. **管线要么原子要么守卫**:计算-断言-落盘分离,破坏性步骤只在前置
   成功后执行。中途断言退出 + 无守卫的后续破坏步,是这次两步错里
   更危险的那一步。
3. **审计产物要落盘**(f401_audit.json):90/62 的分类是可复查的证据,
   不是一次性的内存判断。

产物:~70 个文件的导入区收敛;基线 153 → 1;审计三迭代与两次回滚全程
记录在案。下一版 v1293 修 data_prep 潜在 NameError(带回归测试),
之后 v1294 上优雅指标 ratchet、v1295+ 开始批量重命名+子包化。
