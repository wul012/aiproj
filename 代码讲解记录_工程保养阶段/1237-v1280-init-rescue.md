# 1237 - v1280 init rescue：小初始化杀死 grokking，是一个学习率假象

## 这一版回答什么问题

v1279 的头条发现：把 d=128 的初始范数缩到 d=32 的水平（α=0.5 与匹配范数 α\*），
grokking 被彻底消灭（0/6，100k 步预算）。但这一版的 P1 探针先纠正了 v1279 自己的
解读错误（披露于 `784f91c2`）：那些"死格"其实**瞬间完成了记忆**——final_train=1.000、
t_mem=200–300，和健康格一样快——然后在完全记忆的状态里停满 100k 步。死掉的不是
优化，而是**记忆→泛化的转变**。

于是最锋利的通缩假说浮出水面：**学习率假象**。AdamW 的更新幅度 ~lr 与梯度尺度
无关，所以相对更新量 lr/|w| ∝ 1/α——在 α=0.5 处，冻结的 lr=1e-3 实际上是基线
两倍的相对步长。如果 grokking 的转变需要匹配的有效学习率，重新缩放 lr 就应该救活
它。如果 v1279 的头条死在这里，我们大声说出来（判决阶梯里专门有一支
`norm_clock_revived_under_lr_scaling`，触发即正式降级 v1279 的结论并回写更正）；
如果它扛过 16 倍范围的 lr 扫描，"小初始化阻止 grokking"就从配方运气升级为稳健事实。

## P1 校准（CPU，免费，先于预注册；已披露）

- cache 法证（驱动了 v1279 更正）：全部 8 个删失格 final_train=1.000、
  t_mem=200–300（含 d=64 α=1 的两个死 seed）。
- 初始几何（d=128，seed 1337）：α ≤ 1 时初始 loss ≈ ln(99)（均匀预测）；logit std
  按 ~α² 缩放（0.019/0.080/0.256/0.968 @ α=0.25/0.5/1/2）；梯度范数按 ~α 缩放
  （0.13/0.32/1.08/5.01）。配合 Adam 归一化 ⇒ 相对步长 ∝ 1/α——这确定了 lr 扫描
  以**向下**为主攻方向（向上为诚实对照）。
- d=64（α=1）在初始几何上毫无异常（logit std 0.19 平滑插值于 32 与 128 之间）：
  v1279 的中宽度洞在初始处不可见。仅记录，本版不碰 d=64。

## 实验设计：两条臂 + 只读参照

全部新格 d=128、单一时钟 max_steps=60000（基线中位 t_gen 11,400 的 5 倍以上，
也超过见过的最晚 grok 格 35,000）；其余配方冻结。

- **救援臂**：α=0.5 × lr ∈ {0.25, 0.5, 2, 4}×1e-3 × seeds {1337,1338} = 8 runs。
  α=0.5 lr=1e-3 的参照格直接读 v1279 提交进仓库的 cache（复用缓存规则，绝不重跑）。
- **确认规则（确定性）**：恰好 1/2 grok 的 lr 追加 seed 1339 确认，按 lr 升序，
  预算封顶；rescued(lr) = 该 lr 的 seeds 中 ≥2 个 grok。
- **剂量臂**：α ∈ {0.6, 0.7, 0.85} × lr=1e-3 × seeds {1337,1338} = 6 runs，与
  v1279 参照（α ∈ {0.493, 0.5, 1, 2}）合起来夹住死亡边界。
- 判决阶梯（预注册 `2cd7590c`）：无 lr 获救 → `stuck_memorized_robust_to_lr`；
  有 lr 获救且最佳救援统计量（获救 lr 内 grok 格的中位 t_gen 取最小）< 基线中位
  → `norm_clock_revived_under_lr_scaling`；≥ 基线中位 →
  `lr_rescues_grokking_without_speedup`；G0 = v1279 参照 cache 完整性（α=1 3/3
  grok、中位恰为 11,400、6 个死格全部 stuck_memorized）；G2 = 判决在
  bar ∈ {0.85, 0.90, 0.95} 下逐字一致。剂量臂 cliff/graded 分类纯描述。
- 这一版的格子分类比 v1279 细：grokked / stuck_memorized / broken——broken
  （连记忆都没有）专门用来防止把 lr=4e-3 的训练爆炸误读成"顽固卡死"。cache 这次
  同时保存 train 与 val 曲线，Phase B 能在任意 bar 下重算 t_gen 和 t_mem。

## 实现要点

`src/minigpt/grok_init_rescue_v1280.py` 从 v1279 模块直接 import `scaled_init` /
`t_gen_at` / `total_norm_of`（不复制）；训练仍走 v1185 `train_to_grok`（lr 与
max_steps 都经由现成的 GrokConfig 字段，训练循环零改动）。测试 9 项：三分类
（含 heldout 门与 broken 支）、t_mem 曲线语义、配置校验、G0 对合成参照的
通过/失败、decide() 全部分支（含确认规则把 1/2 提升为 2/3、确认失败回落）、
注入假 trainer 的确认编排、字节稳定契约、图像落盘。预注册前用真实 v1279 cache
跑了一次 G0=True 的完整性验证（CPU 只读）。

## 结果：救活了——但方向和理论预测完全相反

P2 探针先给了第一个意外：理论上最被看好的"步长对称"格（lr 减半到 5e-4）**没有
获救**——t_mem=400 瞬间记忆，然后 60k 步卡死（heldout 0.157），和冻结 lr 下的死格
一模一样。全网格出来后图景彻底清晰：**向下调 lr 的四格全部卡死**（lr=2.5e-4 与
5e-4，t_mem 400–600，记忆完好、转变永不发生）；**向上调 lr 的四格全部获救**——
lr=2e-3 两个 seed 在 3,800/4,000 步 grok（heldout 0.977–0.981），lr=4e-3 更是在
**1,300/2,300 步** grok，heldout **0.9992**：整条研究弧里最快、最准的 d=128 格。
确认规则零触发（每个 lr 都是 0/2 或 2/2），预算 14/18。

预注册阶梯落在 **`norm_clock_revived_under_lr_scaling`**：rescued_lrs=[2e-3,
4e-3]，最佳救援统计量 1,800 < 基线中位 11,400，G0/G1/G2 全过、三个 bar 下判决
逐字一致。按预注册的强制义务，v1279 简报顶部已加链接式降级通告：v1279 的头条
"缩小初始范数阻止 grokking"是**学习率条件性的**——在冻结 lr 下为真，跨 lr 不成立。

剂量臂（描述性）给出冻结 lr 下的死亡形态：**cliff**——α ∈ {0.6, 0.7} 四格全部
卡死，α=0.85 两格全 grok 但已慢 3–4 倍（44,200/35,600；60k 时钟恰好救下 44.2k
那格，v1279 re-panel 的教训直接兑现）。边界在 (0.7, 0.85]，且接近边界时转变先
减速再消失——像一个随 α 收窄直至关闭的"可达窗口"。

## 机制含义与下一个问题

死亡机制不是"相对步长失配"的对称故事：若卡死源于 α=0.5 下相对步长（lr/|w| ∝
1/α）过大，减半 lr 应当修复，实际毫无作用；反而要把相对步长放大到基线的 4–8 倍
才触发转变。更贴切的图像：**离开记忆解的吸引域需要足够大的有效探索步长**，小
初始化在冻结 lr 下探索不足，于是永远停在记忆盆地里。严格说 v1280 证明的是"lr
放大能让小 init 快速 grok"，尚未分离"范数小"与"相对步长大"哪个是速度的真因——
缺的对照是 α=1 配 lr 2–4×（banked，天然是下一版）。
