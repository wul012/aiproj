# 1253 - v1296 正式批 2:receipt_index 链家族最长 30 个模块短名化

## 本版目标与不做什么

优雅计划批量重命名的第一个**正式批**(v1294 试点模式的首次放量):
receipt_index 链家族——222 个模块,名字随版本代际逐层堆积
`receipt_index_` 片段(78 → 88 → 92 → … 每四个版本长一节,直到 193 字符)
——取**最长的 30 个**(193–186 字符,review/check 两个子族),原地重命名
为 `receipt_chain_<role>_<version>` 短规范名,旧长名全部留 sys.modules
转发 shim;90 个消费者文件(脚本+测试)的 import 全部改写到短名。
不做什么:模块**内部**的历史长符号名不动(它们是家族公共 API,由
name-budget 基线管辖、由 v1295 换底工具按"可证明中立"过闸);消费者
脚本/测试自己的长文件名不动(本家族的脚本名与模块名 token 顺序不同,
不在本批机械可推导范围,后续批处理);产物文件名常量一字不动。

## 执行与两个当场修正

批处理脚本沿用试点模式(先全部断言、后统一落盘;git mv 保历史):
30 模块 + 30 shim + 90 消费文件改写一次成型。两个当场修正:

1. **短名自身超预算**:name-budget 连**文件名**也算违规对象(40 字符
   上限)。机械 tail 折叠给 check 子族生成了
   `receipt_chain_receipt_check_vNNNN_artifacts`(46 字符,token 重复)
   ——换底工具**拒绝**换底(new 160 ≠ resolved 150,10 条真新增),
   把这个错误当场按住。补第二遍去重命名
   (`receipt_chain_check_vNNNN_artifacts`,35 字符),shim 目标与
   消费 import 同步重指,换底随即通过:**7,515 → 7,515,零净变**。
   教训:换底工具的拒绝分支第一次实战开火,拦下的正是"新名字本身
   不合格"这种最容易溜过去的错误——工具比人可靠。
2. **消费者脚本/测试 0 重命名是对的**:本家族脚本名前缀化
   (`review_randomized_…_v1050`)与模块名后缀化(`…_review_v1050`)
   token 顺序不同,文件名嵌入推导不命中——批脚本如实报 0,不强行
   猜测;它们的 import 改写(90 文件)不受影响。

## 工程教训

1. **换底工具的价值在拒绝分支**:接受分支只是省事,拒绝分支拦真错。
   10 个超预算新名若溜进基线,就是给未来所有批次开"新长名合法化"
   的先例。
2. **机械推导要敢于返回"不命中"**:消费者文件名的 token 顺序差异
   如果用模糊匹配硬凑,会产生一批错误重命名;0 命中 + 留给后续批,
   比 90% 正确的猜测安全得多。
3. **批的收口从此有标准清单**:map 断言 → 应用 → 换底(拒绝即修名)
   → 家族测试 → 优雅基线重冻结 → 15 闸门全扫 → 全量套件 → 提交。
   本批全程走完,后续批照抄。
