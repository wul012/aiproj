# 1257 - v1300 优雅工程收官:复评 6 → 7.5,账目全公开

## 本版目标与不做什么

优雅工程(v1290-v1299,十个版本)的关账版:出具**可复现的复评报告**
(docs/elegance/program-closeout-v1300.md),对照最初 6/10 评分的同一
把尺子重新量,证据矩阵逐轴给出"当时 → 现在 → 复现命令"。不做什么:
不加新代码;不为凑分做任何最后一刻的化妆性改动;不把"过程的漂亮"
算进"制品的漂亮"——评的是仓库本身,不是这十个版本干得多利索。

## 复评结论:7.5,不是 9——差距的构成与定价

用户的目标是"加约 3 分到 9"。诚实的结论是 **6 → 7.5**:

**拿到的分**:文件名轴彻底清偿(282 个 >120 字符长名归零,最长 202 →
120,CI 从此拒收新长名);静态债 271 → 0 且钉死;12 处复制的 `_median`
归一;360 个转发 shim(282 模块 + 78 脚本)全部有身份测试;覆盖率
89.29 站在 88.98 地板上方;四项优雅度量全部冻结为只紧不松的棘轮。

**没拿到的分**:1,355 个模块仍然平铺(v1294 试点证明 owner-package
架构契约**正确地**挡住了把历史模块原样搬进子包——那是归一化线的活,
搬家不等于达标);7,515 个超预算**符号名**只换了底、从未清偿(改的是
文件名,函数名还是一百多字符);227 个重复函数体只是被冻结、没有烧掉。
这三项就是 7.5 到 9 的价目表,每一项都是独立的长弧,没有便宜货。

当初 8.5-9 的预估是在 v1294 发现架构闸门之前给的;v1294 的讲解当场
记录了收窄。收官报告把这笔账摆在明面上:**预估错在哪、为什么、剩余
分数值多少钱**——这比一个凑出来的 9 有用得多。

## 尺子本身也是产物

复评的每个数字都给了再生成命令(优雅棘轮、静态门、名字预算门、
覆盖率门、shim 测试)。"分数"不再是一次性的主观判断,而是任何人
任何时候可以重跑的测量——这正是六个版本前埋优雅棘轮的目的:先让
度量存在,再谈改进,最后评分只是读表。

## 有意跳过的项也入账

原计划草案里的"拆 1,398 行测试文件"(现 1,309 行)判定不拆:文件
尺寸棘轮管着它,内容是一个连贯契约家族,拆开只增加 import 面、不改
善任何度量或可读性。判断性跳过,记录在案——跳过不是遗漏,是决策。

## 工程教训(全程十版的三条)

1. **棘轮先于改进**:v1293 先把四个度量冻成基线,v1294-v1299 的每
   一步才有"只能变好"的地板。反过来做,清债会和新债赛跑。
2. **契约会正确地挡住你**:v1294 的子包化被 owner-package 契约拦下
   不是障碍,是制度在工作——绕过它(裸搬)才是真正的失败模式。
   诚实收窄范围、把差距定价,比强行达标干净。
3. **收官即定价**:关账报告的价值不在分数,在"剩余每一分值多少钱"
   ——归一化迁移、符号级改名、重复烧减三条长弧,留给下一个程序
   立项时按价取货。
