# v371 maturity benchmark requirement carryover 运行说明

## 本版目标

v371 把 v370 registry 已经聚合出的 benchmark readiness requirement change 继续带入 maturity summary 和 maturity narrative。目标是让成熟度总览也能看到 requirement 从稳定转为失败的风险，而不是只看 benchmark-history regression count。

本版不重新训练模型，不改变 benchmark history、registry 或 release readiness comparison 的原始生成规则，也不把教育型 MiniGPT 误说成生产模型。它只消费已有 registry release-readiness delta，并把 requirement change 提升为 maturity review 信号。

## 运行证据

- `runs/__tmp_v371_maturity/project/runs/maturity-summary/maturity_summary.json`
  - 由 `scripts/build_maturity_summary.py` 生成的临时验证产物。
  - `summary.release_readiness_benchmark_requirement_status_changed_count=1`。
  - `summary.release_readiness_max_benchmark_requirement_exit_code_delta=1`。
  - `summary.release_readiness_trend_status=benchmark-regressed`。
- `runs/__tmp_v371_maturity/project/runs/maturity-narrative/maturity_narrative.json`
  - 由 `scripts/build_maturity_narrative.py` 生成的临时验证产物。
  - 顶层 `portfolio_status=review`，证明 requirement change 会把组合叙事推入复核。
- `d/371/图片/01-maturity-benchmark-requirement.png`
  - Playwright 截图，证明 HTML 报告真实渲染后显示 `Benchmark req changes=1` 和 `Benchmark req exit=1`。
- Playwright 运行期间还做了 snapshot、文本检查和本地 HTTP 预览校验；这些只用于本轮验证，不作为长期归档保留。

## 命令摘要

```text
python -B scripts/build_maturity_summary.py --project-root runs/__tmp_v371_maturity/project --registry runs/__tmp_v371_maturity/project/runs/registry/registry.json --request-history-summary runs/__tmp_v371_maturity/project/runs/request-history-summary/request_history_summary.json --out-dir runs/__tmp_v371_maturity/project/runs/maturity-summary
```

关键 stdout：

```text
release_readiness_trend_status=benchmark-regressed
release_readiness_benchmark_requirement_status_changed_count=1
release_readiness_max_benchmark_requirement_exit_code_delta=1
```

```text
python -B scripts/build_maturity_narrative.py --project-root runs/__tmp_v371_maturity/project --maturity runs/__tmp_v371_maturity/project/runs/maturity-summary/maturity_summary.json --registry runs/__tmp_v371_maturity/project/runs/registry/registry.json --request-history-summary runs/__tmp_v371_maturity/project/runs/request-history-summary/request_history_summary.json --benchmark-scorecard runs/__tmp_v371_maturity/project/runs/benchmark-scorecard/benchmark_scorecard.json --benchmark-scorecard-decision runs/__tmp_v371_maturity/project/runs/benchmark-scorecard-decision/benchmark_scorecard_decision.json --benchmark-history runs/__tmp_v371_maturity/project/runs/benchmark-history/benchmark_history.json --dataset-card runs/__tmp_v371_maturity/project/runs/dataset-card/dataset_card.json --out-dir runs/__tmp_v371_maturity/project/runs/maturity-narrative
```

关键 stdout：

```text
portfolio_status=review
release_readiness_trend_status=benchmark-regressed
release_readiness_benchmark_requirement_status_changed_count=1
release_readiness_benchmark_requirement_exit_code_delta_max=1
```

## 截图说明

`01-maturity-benchmark-requirement.png` 展示的是 maturity narrative HTML 首屏。顶部 stat cards 中 `Portfolio=review`、`Release trend=benchmark-regressed`、`Benchmark req changes=1` 和 `Benchmark req exit=1` 同时出现，说明 maturity 叙事不再漏掉 requirement 变化。

Release Quality Trend 段落进一步写出 `benchmark requirement changes=1` 和 `benchmark requirement exit delta=1`，让审阅者知道这不是普通 benchmark case regression，而是 readiness requirement 自身发生了失败或阈值退化。

## 清理边界

`runs/__tmp_v371_maturity` 是本版为了生成 summary、narrative 和截图而创建的临时输入目录，最终提交前会删除。`d/371` 中的截图、命令输出和说明是最终归档证据，会保留。

一句话总结：v371 把 benchmark readiness requirement 变化推进到 maturity summary 和 narrative 层，让成熟度总览也能把这类风险判为 review。
