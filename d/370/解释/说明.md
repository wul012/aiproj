# v370 registry benchmark requirement 运行说明

## 本版目标

v370 把 v369 已经写入 release readiness comparison 的 benchmark readiness requirement 继续带入 registry。目标是让多运行目录的汇总页面、CSV 和 CLI 日志直接暴露 requirement 从 `pass` 变为 `fail` 的风险。

本版不重新训练模型，不改变 benchmark history 的生成规则，也不改变 release readiness comparison 的原始判断。它只消费已有 comparison delta，并把字段提升到 registry 层。

## 运行证据

- `runs/registry-v370/registry.json`
  - 由 `scripts/register_runs.py` 生成的 registry 主证据。
  - `release_readiness_delta_summary` 中包含 `benchmark_history_readiness_requirement_status_changed_count=1` 和 `max_abs_benchmark_history_readiness_requirement_exit_code_delta=1`。
- `runs/registry-v370/registry.csv`
  - 新增 run 级字段 `release_readiness_benchmark_requirement_status_change_count` 和 `release_readiness_benchmark_requirement_exit_code_delta_max`。
- `runs/registry-v370/registry.html`
  - 浏览器页面显示 `benchmark req changes=1 exit max=1`。
  - Release Readiness Deltas 表格显示 `req=pass -> fail exit=+1`。
- `d/370/图片/01-registry-benchmark-requirement.png`
  - Playwright 截图，证明 HTML 报告真实渲染并能看到 requirement 变化。
- `d/370/解释/playwright-check.txt`
  - Playwright 文本检查返回 `true`，验证页面同时包含 run 级和 delta 级 requirement 诊断。

## 命令摘要

```text
python -B scripts/register_runs.py runs/__tmp_v370_registry/baseline runs/__tmp_v370_registry/candidate --name baseline --name candidate --out-dir runs/registry-v370
```

关键 stdout：

```text
release_readiness_benchmark_history_regressions=1
release_readiness_benchmark_requirement_status_changes=1
release_readiness_benchmark_requirement_exit_delta_max=1
```

## 截图说明

`01-registry-benchmark-requirement.png` 证明 registry 页面已经不只显示 benchmark-history regression 数量，还显示 requirement status change 和 exit delta。这样审阅多次运行时，不需要逐个打开 readiness comparison JSON，也能看到 benchmark readiness requirement 的失败。

## 清理边界

本版 `runs/__tmp_v370_registry` 只是生成 registry 的临时输入，最终提交时不保留。`runs/registry-v370` 只作为验证中间产物，也不作为最终版本证据提交；归档证据保留在 `d/370`。

一句话总结：v370 把 benchmark readiness requirement 的失败从单个 readiness comparison 推到 registry 多运行审阅层。
