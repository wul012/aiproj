# v363 governance routing gate

## 本版主题

v363 在 v362 的基础上再往前走一步：把治理提案路由的“clean/blocked”判断变成一个可选门禁。

这版的核心不是增加更多路由规则，而是把已经存在的 `exact / keyword / ambiguous_keyword / review / reject` 路由结果，包装成一个 CI 可直接判断退出码的门槛：

- 默认只报告，不阻断
- 传 `--require-clean-governance-routing` 时，遇到 `review_before_merge` 或新链候选就退出 1

## 本版来自哪里

这一版承接的是 v357-v362 的治理收口路线：

- v357 冻结新链扩展
- v358 要求每条链有 review reason 和 expansion rule
- v359 把 proposal 路由到已有链
- v360 记录 exact/keyword basis
- v361 记录具体 keyword hit
- v362 对多链命中的 proposal 强制进入 review
- v363 把“是否允许 clean merge”做成显式门禁

## 关键文件

- `scripts/check_maintenance_batching.py`
  - 新增 `--require-clean-governance-routing`
  - 输出 routing requirement 状态、决策、exit code、blocking count 和 failed reasons
  - 在门禁失败时退出非零
- `src/minigpt/maintenance_policy_artifacts.py`
  - JSON / Markdown / HTML 现在都带 `routing_requirement`
  - HTML 新增 `Routing Requirement` 区块，便于截图审计
- `tests/test_maintenance_policy.py`
  - 新增 clean 通过和 ambiguous 阻断两条脚本测试
  - 保证门禁不是只在 stdout 里看起来成立

## 核心数据结构

### `routing_requirement`

这是 v363 新增的门禁摘要，主要字段是：

- `required`: 是否启用门禁
- `status`: `not-required` / `pass` / `fail`
- `decision`: `report-only` / `continue` / `stop`
- `exit_code`: `0` 或 `1`
- `blocking_count`: 需要人工处理的 proposal 数量
- `failed_reasons`: 例如 `review_required`、`ambiguous_keyword`、`new_chain_candidate`

它是发布前的判断层，不是治理链本体。

## 运行流程

1. CLI 读 governance proposals。
2. `build_governance_stabilization_review()` 先生成路由和摘要。
3. `build_governance_routing_requirement()` 根据路由结果判断是否需要拦截。
4. 该门禁结果写入 JSON / Markdown / HTML。
5. stdout 打印门禁状态。
6. 如果启用了 `--require-clean-governance-routing` 且结果失败，CLI 退出 1。

## 产物说明

- `governance_stabilization.json`
  - 最终机器可读证据。
  - 同时包含 proposal routing 和 routing requirement。
- `governance_stabilization.md`
  - 人工审阅版证据。
  - 适合快速看“为什么会 stop”。
- `governance_stabilization.html`
  - 浏览器截图证据。
  - v363 截图归档在 `d/363`。

## 测试覆盖

新增测试保护了这些行为：

- clean proposal 组合在门禁开启时仍然退出 0
- ambiguous proposal 在门禁开启时退出 1
- JSON / Markdown / HTML 都带 `routing_requirement`
- CLI stdout 和最终输出一致

## 一句话总结

v363 把治理提案路由从“能识别 ambiguous”推进到“能在 CI 里按需阻断 ambiguous”，让收口期的扩链更可控。
