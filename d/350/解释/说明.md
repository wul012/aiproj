# v350 运行截图说明：registry benchmark readiness tracking

本版目标是把 v349 release readiness comparison 里的 benchmark history regression 继续带入 run registry。它不重新比较 readiness，不训练模型，也不改变 release gate；只让多 run 索引也能看到 benchmark history 是否跨版本退化。

## 截图清单

- `01-registry-benchmark-history-readiness-html.png`：Playwright/Chrome 打开本版生成的 `registry.html`，证明 registry 顶部出现 `benchmark-regressed:1`，`Benchmark regressions` 卡片显示 `regressions=1 case=2 flags=1 boundary=1`，Release Readiness Deltas 表出现 `Benchmark history` 列。
- `02-verification-summary.png`：本版验证摘要，记录 registry 聚焦测试、语法检查、CLI smoke、Playwright 截图，以及后续全量验证项。

## Smoke 边界

本版 smoke 构造两个 registry run：

```text
baseline-run: release readiness improved
candidate-run: benchmark_history pass -> warn
```

candidate 的 benchmark history delta 是：

```text
benchmark_history_status_delta=-1
benchmark_history_ready_delta=-1
benchmark_history_case_regression_delta=2
benchmark_history_generation_flag_regression_delta=1
benchmark_history_latest_boundary_changed=True
```

预期 registry 输出：

```text
release_readiness_comparison_counts={"improved": 1, "benchmark-regressed": 1}
release_readiness_benchmark_history_regressions=1
release_readiness_coverage_regressions=0
release_readiness_ci_order_regressions=0
```

这说明 registry 能独立呈现 benchmark-history readiness regression，不会误归入 CI 或 coverage。

一句话总结：v350 让 benchmark history 的 readiness comparison 退化进入 registry 多 run 索引。
