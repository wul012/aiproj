# v353 dataset version comparison 运行截图说明

## 本版目标

v353 把 v352 的单个 `dataset_version.json` snapshot 推进为多版本比较证据。它解决的问题是：准备了 raw、deduped 或后续更大语料版本之后，reviewer 能不能先确认数据本身发生了什么变化，再去解释训练或 benchmark 的变化。

本版不训练模型，不比较 checkpoint，也不把数据变化解释成模型质量变化。它只负责把 dataset identity、fingerprint、dedupe policy、source order digest、included/skipped source counts 和字符数差异放到同一个报告里。

## 运行链路

本次 smoke 使用同一组源文件生成两个准备数据版本：

- `demo-v353@raw`：不开启去重，3 个源文件全部进入 corpus。
- `demo-v353@deduped`：开启 `--dedupe-exact-sources`，3 个源文件中 2 个进入 corpus，1 个重复源文件被跳过。

随后运行：

```powershell
python -B scripts/compare_dataset_versions.py `
  runs/v353-dataset-version-comparison-smoke/datasets/demo-v353/raw `
  runs/v353-dataset-version-comparison-smoke/datasets/demo-v353/deduped `
  --name raw --name deduped --baseline raw `
  --out-dir runs/v353-dataset-version-comparison-smoke/comparison
```

输出包括：

- `dataset_version_comparison.json`
- `dataset_version_comparison.csv`
- `dataset_version_comparison.md`
- `dataset_version_comparison.html`

这些比较文件是只读证据，源头仍然是两个 prepared dataset 目录中的 `dataset_version.json`。

## 截图说明

### `图片/01-dataset-version-comparison.png`

这张图是新生成的 dataset version comparison HTML。它证明：

- baseline 是 `raw`。
- 一共比较 2 个 dataset version。
- fingerprint 数量是 2，说明 corpus 内容发生变化。
- dedupe policy 数量是 2，说明 raw 与 deduped 的准备策略不同。
- deduped 版本相对 raw 出现 `fingerprint, dedupe, source-order, chars` 变化。
- deduped 的 included source 从 3 降到 2，skipped source 从 0 升到 1，char count 从 140 降到 90。

### `图片/02-deduped-dataset-version-snapshot.png`

这张图是 deduped 单版本的 `dataset_version.html`。它证明比较报告不是凭空推断，而是消费 v352 已经落地的 snapshot 证据：

- dataset id 是 `demo-v353@deduped`。
- dedupe policy 是 `exact-source-content`。
- sources 是 3，included 是 2，skipped 是 1。
- fingerprint 是 `8af2972fba7f`。

## 验证覆盖

本版验证分三层：

- 单元测试覆盖 `summarize_dataset_version()`、baseline delta、renderer escaping、facade export 和输出文件写入。
- smoke 命令真实生成 raw/deduped 两个 prepared dataset，再运行比较 CLI。
- Playwright 打开 HTML 产物并截图，确认报告在浏览器里可读。

## 一句话总结

v353 让 MiniGPT 的数据治理从“单个数据版本有 snapshot”推进到“多个数据版本能横向比较并解释数据变化边界”。
