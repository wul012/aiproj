# v447 运行截图和解释

## 本版主题

v447 把 release readiness comparison 已经识别出的 CI boundary plan-check regression 继续向下游传递。v446 能在 ready-vs-ready 的比较里发现 `boundary_gate_plan_check_not_ready`；本版解决的是另一个断点：这个 regression count 不能只停留在 comparison artifact，而要进入 registry、maturity summary、maturity narrative 和 training portfolio comparison review action。

本版不新增治理链，不重新训练模型，也不改变 release readiness 自身的判定。它只是把已有 CI 契约回归计数向后传递，让后续成熟度和组合评审不会漏掉最佳分数候选身上的 CI boundary plan-check 问题。

## 运行证据

- `registry-fixture/boundary-plan/`
  - 受控 run fixture，包含最小 checkpoint、manifest、dataset、eval、generation quality、benchmark scorecard 和 release-readiness-comparison。
  - release readiness comparison 的整体状态保持可读，但 delta 中 `ci_workflow_baseline_candidate_threshold_boundary_gate_plan_check_ready_regressed=True`。
- `registry/`
  - `register_runs.py` 生成的 registry JSON/CSV/SVG/HTML。
  - `release_readiness_delta_summary` 中保留 `ci_workflow_baseline_candidate_threshold_boundary_gate_plan_check_ready_regression_count=1`。
- `maturity-summary/`
  - `build_maturity_summary.py` 生成的 maturity summary。
  - CLI stdout 显示 `current_version=447` 和 `release_readiness_ci_boundary_plan_check_ready_regression_count=1`。
- `maturity-narrative/`
  - `build_maturity_narrative.py` 生成的 narrative。
  - Summary 将同一计数带入发布质量叙事层。
- `portfolio-fixture/`
  - baseline/candidate 两个 training portfolio fixture。
  - candidate 分数更高，但 maturity narrative 携带 `boundary_gate_plan_check_not_ready`。
- `training-portfolio-comparison/`
  - `compare_training_portfolios.py` 生成 JSON/CSV/Markdown/HTML。
  - best-score candidate 被 review action 标成 blocker，原因是 release-readiness CI boundary plan-check regression。
- `training_portfolio_comparison_snapshot.md`
  - Playwright MCP 对 HTML 页面的可访问性快照。

## 关键结论

这组证据证明：v447 不是重新定义 release readiness，而是把 v446 暴露出的 CI boundary plan-check 回归传到组合评审层。即便 candidate 的 benchmark 分数更高，portfolio comparison 仍会因为 `boundary_gate_plan_check_not_ready:1` 给出 blocker action。
