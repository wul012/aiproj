# v460 运行截图和解释

## 本版主题

v460 给 v459 的 `run_ci_promoted_seed_receipt_contract_failure_smoke.py` wrapper 增加一层 plan check。
v459 已经能把 receipt contract summary 和 failure smoke 封装成一个 CI wrapper，并写出 invocation plan；v460 继续确认这个 plan 没有被篡改、没有丢 artifact、没有把失败子命令伪装成通过。

本版不新增新的训练能力，也不改变 receipt failure smoke 的四个场景矩阵；它只补上 wrapper 产物的可复核入口，并把这个入口接入 GitHub Actions 和 CI workflow hygiene。

## 关键产物

- `receipt-failure-smoke-wrapper/`
  - v459 wrapper 的实际输出。
  - 其中 `ci_promoted_seed_receipt_contract_failure_smoke_plan.json` 是本版 plan check 的输入。
- `receipt-failure-smoke-plan-check/`
  - v460 新增检查输出。
  - 包含 `ci_promoted_seed_receipt_contract_failure_smoke_plan_check.json/txt/md/html`。
- `ci-workflow-hygiene/`
  - 验证 GitHub Actions 已经按 wrapper -> plan check -> coverage 的顺序执行。
- `playwright_receipt_wrapper_plan_check_snapshot.md`
  - Playwright MCP 页面快照。
- `图片/01-receipt-wrapper-plan-check.png`
  - Playwright MCP 对 plan-check HTML 的完整页面截图。

## Plan Check 证明什么

本版检查项覆盖：

- `wrapper` 必须是 `run_ci_promoted_seed_receipt_contract_failure_smoke`。
- `status=pass`，`decision=receipt_contract_failure_smoke_verified`。
- `receipt_contract_summary` 和 `receipt_contract_failure_smoke` 两个 child command 的 return code 都是 `0`。
- `failure_smoke_summary` 必须保留 `available=True`、`status=pass`、`scenario_count=4`、`verified_scenario_count=4`、`failed_verification_count=0`。
- `contract_summary_json`、`failure_smoke_json`、`failure_smoke_csv`、`failure_smoke_html` 必须存在，并且当前文件 SHA-256 与 plan 记录一致。

这说明 v459 wrapper 的 plan 不是孤立日志，而是能被后续 CI step 重新验证的 contract artifact。
