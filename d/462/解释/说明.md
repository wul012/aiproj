# v462 运行截图和解释

## 本版目标

v462 延续 v461 的 receipt failure-smoke plan-check readiness 字段，但不再向 audit/bundle/readiness 继续堆 carryover，而是把它接入已有的 release readiness comparison 链路。

本版解决的问题是：两个 release readiness 报告都可能仍是 `ready`，但 current 把 `ci_workflow_promoted_seed_receipt_contract_failure_smoke_plan_check_ready` 从 `True` 变成 `False`。如果比较层不识别这个变化，发布侧会误以为两版没有 CI 证据退化。

## 运行证据

- `source-inputs/`：保留本版复现实验输入，两份 release readiness JSON 分别代表 baseline 和 current。
- `release-readiness-comparison-receipt-plan/`：由 `scripts/compare_release_readiness.py` 生成的 JSON/CSV/delta CSV/Markdown/HTML。
- `playwright_receipt_plan_comparison_snapshot.md`：Playwright MCP 可访问性快照，确认 HTML 中显示 `CI receipt plan changes=1`、`CI receipt plan regressions=1`、`receipt_failure_smoke_plan_check_not_ready:1`。
- `../图片/01-receipt-plan-comparison-regression.png`：Playwright MCP 截图，确认 comparison HTML 可视化渲染正常。

## 截图说明

`图片/01-receipt-plan-comparison-regression.png` 展示的是 release readiness comparison HTML。页面 summary 区显示：

- `Ready=2`
- `Regressed=0`
- `CI regressions=1`
- `CI receipt plan changes=1`
- `CI receipt plan regressions=1`
- `CI regression reasons=receipt_failure_smoke_plan_check_not_ready:1`

这说明 v462 的判断边界是合理的：整体 readiness 没有下降，但 CI 证据链发生了退化，因此只标记 CI regression，不把 overall readiness delta 误报成 release regression。

## 关键结论

- baseline：`ci_workflow_promoted_seed_receipt_contract_failure_smoke_plan_check_ready=True`
- current：`ci_workflow_promoted_seed_receipt_contract_failure_smoke_plan_check_ready=False`
- delta：`ci_workflow_promoted_seed_receipt_contract_failure_smoke_plan_check_ready_regressed=True`
- reason：`receipt_failure_smoke_plan_check_not_ready`

一句话总结：v462 让 release readiness comparison 能识别 receipt plan-check readiness 的隐性退化，即使两个版本的最终 readiness 仍然都是 ready。
