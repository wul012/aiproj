# v359 governance proposal routing 运行截图说明

## 本版目标

v359 是 v357-v358 governance stabilization 收口线的继续版。v357 先暂停新增治理链，v358 再把每条链的 `review_reason` 和 `expansion_rule` 写清楚；v359 继续保持“不新增第 8 条治理链”的边界，但把新增提案也接入同一条判断线，先尝试路由到现有 7 条链，再决定是否需要人工复核。

本版解决的问题是：当新的治理想法出现时，不要直接把它当成新链候选，而要先判断它能否并入现有链，或者必须先复核再说。

本版不训练模型，不新增独立门禁，也不把治理链扩成新的仪表盘层。它只把“新提案如何落到现有链”这一步补成机器可读证据。

## 运行链路

运行命令：

```powershell
python -B scripts/check_maintenance_batching.py `
  --skip-module-pressure `
  --governance-proposals runs/__tmp_governance_proposals.json `
  --out-dir runs/v359-governance-proposal-routing-smoke
```

新增输出仍然复用治理稳定评审产物：

- `governance_stabilization.json`
- `governance_stabilization.csv`
- `governance_stabilization.md`
- `governance_stabilization.html`

## 截图说明

### `图片/01-governance-proposal-routing.png`

截图证明 governance stabilization review 现在除了 7 条固定链之外，还显示 proposal routing 区块：

- `Routing = merge_existing`

- `Review = 0`

这说明新增提案不是直接堆到新链，而是先归并到现有链，并把结果写进输出证据。

## 验证覆盖

本版验证覆盖：

- `tests.test_maintenance_policy`：检查默认无提案、可并入现有链、高风险提案复核、未匹配提案拒绝。
- Markdown/HTML/CSV/JSON 输出包含 proposal routing 区块。
- CLI stdout 输出 routing 计数。
- Playwright 打开 HTML 并截图归档。

## 一句话总结

v359 把治理稳定评审从“只暂停新增链”推进到“先把新提案路由进现有链”，让收口阶段也能继续接住真实需求。
