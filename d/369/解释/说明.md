# v369 release readiness benchmark requirement evidence

## 本版主题

v369 把 benchmark history readiness requirement 从 release gate 继续带入 release readiness dashboard 和 readiness comparison。目标是让发布准备态不只看到 `benchmark_history_status`，也能看到 requirement status、exit code 和 failed reasons。

本版不改变 benchmark history 的计算规则，也不扩大模型评测规模。它只验证 readiness 层和 readiness comparison 层能消费并比较 requirement 字段。

## 关键截图

- `d/369/图片/01-release-readiness-requirement-comparison.png`

截图来自 `release_readiness_comparison.html`，页面显示：

- baseline readiness 为 `ready`
- current readiness 为 `review`
- benchmark history 仍为 `pass`
- benchmark readiness 从 `pass` 变为 `fail`
- benchmark readiness exit delta 为 `1`
- benchmark regression count 为 `1`
- CI regression 和 coverage regression 均为 `0`

这证明当前回退来自 benchmark readiness requirement，而不是 CI、coverage 或旧的 benchmark status。

## 运行命令

```powershell
python -B scripts\compare_release_readiness.py `
  --readiness runs\__tmp_v369\baseline\release-readiness\release_readiness.json `
  --readiness runs\__tmp_v369\current\release-readiness\release_readiness.json `
  --out-dir runs\v369-release-readiness-comparison
```

关键输出：

```text
readiness=2
baseline_status=ready
regressed=1
ci_workflow_regressions=0
test_coverage_regressions=0
benchmark_history_deltas=1
benchmark_history_regressions=1
benchmark_history_readiness_requirement=v368-baseline:pass:0
benchmark_history_readiness_requirement=v369-current:fail:1
```

## 证据边界

这个证据说明 release readiness comparison 可以识别 benchmark readiness requirement 回退。它不是模型质量证明；模型质量仍需要真实 checkpoint、固定 benchmark、scorecard 和 repeated history 证据。

## 清理说明

用于生成截图的临时 readiness JSON、comparison report 和本地 HTTP server 在任务收尾时删除或停止。`d/369` 只保留最终截图和本说明。
