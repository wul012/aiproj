# v448 运行截图和解释

## 本版主题

v448 把 promoted training scale clean-batch 链路里的 CI boundary plan-check regression 从 comparison 继续传到 decision、seed、seed handoff、automation receipt、embedded receipt check 和 handoff assurance。

本版不新增新的治理链，不重新训练模型，也不把 boundary plan-check 解释成模型能力退步。它只解决一个证据断点：如果 `boundary_gate_plan_check_not_ready` 已经能从 CI reason map 中识别出来，下游 clean-batch 评审和 handoff receipt 也应该保留这条窄语义计数。

## 运行证据

- `fixture/`
  - 受控 promotion index fixture。
  - `beta-boundary-plan` 携带 `handoff_batch_maturity_ci_regression_count=2`，其中 `boundary_gate_plan_check_not_ready=1`。
- `promoted-comparison/`
  - `compare_promoted_training_scale_runs.py` 输出 comparison JSON/CSV/Markdown/HTML。
  - Summary 显示 `handoff_batch_maturity_ci_boundary_plan_check_ready_regression_count=1`。
- `promoted-decision/`
  - `decide_promoted_training_scale_baseline.py` 输出 decision 证据。
  - Selected baseline 是 clean 的 `gamma`，但 rejected input 保留 boundary plan-check regression count。
- `promoted-seed/`
  - `build_promoted_training_scale_seed.py` 输出 seed 证据。
  - Seed 是 review 状态，继续保留 rejected handoff boundary plan-check count。
- `promoted-handoff/`
  - `execute_promoted_training_scale_seed.py` 输出 handoff JSON/CSV/Markdown/HTML 和 automation receipt。
  - Receipt schema version 为 `4`，包含 boundary plan-check count。
- `receipt-check/`、`embedded-receipt-check/`、`handoff-assurance/`
  - 三层检查均为 `status=pass`，证明 receipt sidecar 和主 handoff report 中的 boundary 字段一致。
- `playwright_comparison_snapshot.md` 与 `playwright_handoff_snapshot.md`
  - Playwright MCP 对 HTML 页面的可访问性快照。

## 关键结论

v448 证明 promoted training scale 的 clean-batch 证据不再只知道“有 CI regression”，还知道其中是否包含 boundary plan-check readiness regression。这样 seed handoff 和 automation receipt 能在不打开上游 comparison JSON 的情况下，直接复核该窄语义风险。
