# v433 运行截图说明：baseline-candidate handoff

## 本版目标

v433 承接 v432 的 baseline-candidate eval loop，把“候选是否通过接收条件”继续落成“下一轮 baseline 应该使用谁”的交接产物。它解决的是交接边界：上游 loop 已经能判断 `accept_candidate` 或 `reject_candidate`，但后续训练/评估流程还需要一份稳定、机器可读、人工可审的 handoff artifact 来说明下一轮 baseline、证据来源和失败原因。

本版不重新训练模型，也不新增一条治理链。它复用 `d/432/解释/baseline-candidate-eval-loop/baseline_candidate_eval_loop.json`，只把已有 loop 结果转成 next-baseline handoff。当前 tiny candidate 仍未达到 `min_overall_score_delta=1.0`，因此 handoff 的保守结论是保留当前 baseline。

## 命令摘要

本版核心运行命令：

```text
python -B scripts\build_baseline_candidate_handoff.py d\432\解释\baseline-candidate-eval-loop --out-dir d\433\解释\baseline-candidate-handoff --require-accepted --force
```

该命令按设计返回 `exit_code=2`，因为 v432 loop 本身执行通过，但候选没有达到本轮最小分差要求，不能交接为下一轮 baseline。

关键输出：

```text

decision=keep_current_baseline
handoff_ready=False
next_baseline_name=tiny-baseline
next_baseline_source=current_baseline
next_baseline_checkpoint_exists=True

candidate_accepted=False

execution_source_mode=rerun_smoke

execution_expected_exit_code=2
rejected_reasons=min_overall_score_delta expected >= 1.0, got 0.0
```

## 证据边界

`d/433` 证明 v432 loop 的结果可以被转换成下一轮 baseline handoff，并且在候选未达标时会保守选择当前 baseline。它不证明模型能力提升，也不证明候选应该进入生产；它只是把候选未被接受的原因、当前 baseline checkpoint 和后续动作固定成可审计产物。

一句话总结：v433 把 baseline-candidate eval loop 的拒绝结果推进成可交接的 next-baseline handoff，让后续训练治理知道“继续用谁”和“为什么不能换”。
