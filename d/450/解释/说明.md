# v450 运行截图和解释

## 本版主题

v450 给 promoted seed handoff receipt contract summary 增加机器可读的 `contract_checks`。v449 已经能展示 schema-v4 boundary plan-check scope，但自动化如果要判断具体哪条 contract 失败，仍需要解析 `issues` 字符串。v450 把这些判断拆成稳定 check row。

本版不新增新治理链，不改变 receipt schema，也不改变 handoff 执行逻辑。它只让现有 contract summary 更适合 CI、自动化脚本和后续审计消费。

## 运行证据

- `contract-summary/`
  - 输入为 `d/448/解释/promoted-handoff`。
  - 输出 `promoted_training_scale_seed_handoff_receipt_contract_summary.*`。
  - JSON 中新增 `contract_checks`、`contract_check_count=10`、`failed_contract_check_count=0`。
- `contract-summary-check/`
  - 从 summary 反向重建 expected summary。
  - `status=pass`，证明 `contract_checks` 和 text/Markdown/HTML sidecar 都可复核。
- `contract_summary_stdout.txt`
  - CLI 输出包含 `receipt_contract_checks=[...]`，便于 shell 或 CI 日志直接检查。
- `playwright_contract_checks_snapshot.md`
  - Playwright MCP 对 contract summary HTML 的可访问性快照。

## 关键结论

v450 证明 receipt contract summary 不再只有自然语言 issues 和两个 scope 表，而是能输出稳定的机器检查列表。后续 CI 或跨项目消费方可以直接读取 `contract_checks[].id/status/expected/actual`，不用猜测 issue 文案。
