# v452 运行截图和解释

## 本版主题

v452 给 promoted seed handoff receipt contract summary-check 的结构化比较行补齐元数据。v451 已经有 `summary_field_checks` 和 `sidecar_checks`，但消费方仍需要从 `key`、`id` 或字段组合推断每行是什么。v452 让每行直接说明自己的类型、目标、状态域、是否必需、值类型和解释。

本版不新增治理链，不改变 receipt contract summary 的业务判断，也不改变 CLI 的默认退出语义。它只让既有 summary-check 输出更适合 CI、后续报告和自动化交接消费。

## 运行证据

- `contract-summary/`
  - 输入为 `d/448/解释/promoted-handoff`。
  - 生成 v452 使用的 promoted seed handoff receipt contract summary。
- `contract-summary-check/`
  - 输出 `promoted_training_scale_seed_handoff_receipt_contract_summary_check.*`。
  - JSON 中 `summary_field_check_count=21`，`failed_summary_field_check_count=0`。
  - JSON 中 `sidecar_check_count=3`，`failed_sidecar_check_count=0`。
  - 每个 summary field row 都包含 `id`、`check_type=summary_field`、`target`、`status_domain`、`required`、`expected_kind`、`actual_kind` 和 `detail`。
  - 每个 sidecar row 都包含 `check_type=sidecar_digest`、`target`、`status_domain`、`required`、`expected_kind=sha256`、`actual_kind=sha256` 和 digest。
- `contract_summary_check_stdout.txt`
  - CLI 输出中可以看到新增 row metadata。
- `playwright_summary_check_row_metadata_snapshot.md`
  - Playwright MCP 对 summary-check HTML 的可访问性快照。

## 关键结论

v452 让 summary-check 的每个比较结果不再只是“字段名 + pass/fail”，而是可以被机器直接识别的 check row。后续如果 registry、release gate 或 CI 摘要消费这些 rows，可以按 `check_type` 和 `target` 聚合，不需要解析自然语言或猜测字段含义。
