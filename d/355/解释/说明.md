# v355 registry dataset snapshot carryover 运行截图说明

## 本版目标

v355 把 v354 的 run comparison dataset snapshot 信号继续接入 run registry。此前 registry 只把 dataset fingerprint 当作主要数据标识，reviewer 需要点进 run 目录或 comparison 报告，才能判断数据版本是否经过 dedupe、是否跳过源文件、语料规模是否变化。

本版不新增独立报告，不训练模型，不改变 registry 排名逻辑。它只让 registry 作为多 run 入口时，能直接展示数据边界。

## 运行链路

本次 smoke 构造两个 run：

- `raw`：dataset `demo-v355@raw`，dedupe policy `none`，included 3，skipped 0，chars 140。
- `deduped`：dataset `demo-v355@deduped`，dedupe policy `exact-source-content`，included 2，skipped 1，chars 90。

随后运行：

```powershell
python -B scripts/register_runs.py `
  runs/v355-registry-dataset-snapshot-smoke/raw `
  runs/v355-registry-dataset-snapshot-smoke/deduped `
  --name raw --name deduped `
  --out-dir runs/v355-registry-dataset-snapshot-smoke/registry
```

输出仍然是既有 registry 产物：

- `registry.json`
- `registry.csv`
- `registry.svg`
- `registry.html`

## 截图说明

### `图片/01-registry-dataset-snapshot.png`

截图证明 registry HTML 已经带上 dataset snapshot 语义：

- 顶部 stat card 显示 `Dataset snapshots = snapshots:2, missing:0, skipped-runs:1`。
- 顶部 stat card 显示 `Dedupe policies = exact-source-content:1, none:1`。
- `deduped` 行显示 `dedupe=exact-source-content; included=2; skipped=1; chars=90; order=order-deduped`。
- `raw` 行显示 `dedupe=none; included=3; skipped=0; chars=140; order=order-raw`。
- 页面底部新增 `Dataset Snapshot Summary`，保留机器可读聚合字段。

这说明 reviewer 从 registry 入口就能看到数据变化，而不需要先进入每个 run 的 dataset version 文件。

## 验证覆盖

本版验证覆盖：

- `tests.test_registry`：检查 registry summary、run row、CSV、SVG 和 HTML 都包含 dataset snapshot 字段。
- `tests.test_registry_split`、`tests.test_registry_assets`、`tests.test_registry_rankings`：确保 registry 拆分后的 facade、资产和 leaderboard 行为没有回归。
- smoke：真实生成两个 run 目录并运行 `scripts/register_runs.py`。
- Playwright：打开 `registry.html` 并截图归档。

## 一句话总结

v355 让 registry 从“只列 run 和 fingerprint”的入口，升级为能直接展示数据版本边界的多 run 审计面板。
